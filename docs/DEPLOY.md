# DEPLOY.md

Универсальная инструкция для AI-агента и владельца проекта по безопасному деплою в разные среды: Vercel, Timeweb Cloud, Railway, Render, Docker VPS и другие.

## Цель документа

Этот файл нужен, чтобы деплой можно было запускать без отдельного DevOps-специалиста.  
AI-агент должен читать этот документ перед любым релизом и действовать строго по нему.

## Для каких сред подходит

- Vercel.
- Timeweb Cloud.
- Railway.
- Render.
- Docker на VPS.
- Любая платформа, где есть staging и production или их аналоги.

## Общий принцип

Деплой всегда идёт по маршруту:

1. Локальная проверка.
2. Деплой в тестовую среду.
3. Проверка после деплоя.
4. Только потом деплой в production.
5. После релиза — мониторинг и готовность к откату.

## Окружения

### Local

Локальная среда разработчика.

Используется для:
- запуска проекта на компьютере;
- первичной проверки;
- ручного тестирования;
- проверки переменных окружения.

### Staging

Тестовая среда, максимально похожая на production.

Используется для:
- проверки перед релизом;
- приёмки владельцем;
- smoke-теста;
- проверки интеграций.

### Production

Боевая среда для реальных пользователей.

Правила:
- деплой только после staging;
- деплой только при понятном плане отката;
- никаких экспериментальных изменений напрямую.

## Кто что может делать

### Разрешено без подтверждения

- Проверять статус сборки.
- Читать логи.
- Деплоить в staging.
- Выполнять smoke-тест после staging.

### Только после подтверждения владельца

- Деплой в production.
- Изменение переменных окружения.
- Применение миграций базы данных.
- Переключение домена.
- Откат production.

### Запрещено

- Удалять production-базу данных.
- Менять секреты без фиксации в документации.
- Выполнять необратимые действия без явного подтверждения.
- Деплоить в production, если staging не проверен.

## Как использовать чеклисты

В проекте есть два разных чеклиста, и у каждого своя роль:

- `docs/DEPLOY-CHECKLIST.md` — использовать ПЕРЕД КАЖДЫМ деплоем
  в staging и особенно перед production. Это технический список:
  код, тесты, env‑переменные, миграции, инфраструктура.

- `docs/RELEASE-CHECKLIST.md` — использовать ПЕРЕД ПЕРВЫМ запуском
  продукта для реальных пользователей или перед крупным публичным
  релизом. Это продуктовый список: сценарии пользователя, безопасность,
  UX, документация.

Агент обязан:

1. Перед любым деплоем пройти `DEPLOY-CHECKLIST.md`.
2. Перед «боевым» запуском продукта пройти ещё и `RELEASE-CHECKLIST.md`.
3. Остановить production‑деплой, если критичные пункты чеклистов не выполнены.

## Универсальный сценарий деплоя

### Шаг 1. Проверить готовность

Перед любым деплоем нужно убедиться:

- код закоммичен;
- ветка понятна (`main`, `staging`, `release/*`);
- переменные окружения описаны в `ENV.md`;
- есть актуальный `CHANGELOG.md`;
- есть план отката в `ROLLBACK-PROTOCOL.md`;
- последний `HANDOFF.md` не противоречит релизу.

### Шаг 2. Прогнать проверки

Минимум:

- линтер;
- тесты, если есть;
- сборка проекта;
- ручная проверка критического сценария.

### Шаг 3. Деплой в staging

Варианты в зависимости от платформы:

- **Git push** в staging-ветку.
- **CLI-команда** платформы.
- **Кнопка Deploy** в панели.
- **AI-агент через MCP / SSH / dashboard workflow**.

После деплоя агент обязан зафиксировать:
- куда задеплоено;
- ссылку на окружение;
- статус сборки;
- краткий результат проверки.

### Шаг 4. Smoke-test staging

Проверить:

- открывается главная страница;
- работает вход или основной сценарий;
- нет явных 500/404 ошибок;
- интеграции отвечают;
- критические формы работают;
- нет сломанных env-переменных.

### Шаг 5. Подтверждение владельца

Перед production агент обязан спросить подтверждение в явной форме:

> Staging проверен. Деплоить в production? Да / нет.

### Шаг 6. Деплой в production

Production-деплой выполняется только после явного подтверждения.

После завершения зафиксировать:
- время релиза;
- commit / tag;
- адрес production;
- успешность smoke-теста;
- кто подтвердил релиз.

### Шаг 7. Мониторинг после релиза

Первые 15–30 минут проверить:

- доступность сайта или API;
- логи ошибок;
- скорость ответа;
- ключевой пользовательский сценарий;
- алерты в мониторинге.

## Примеры запуска деплоя по типу платформы

### Вариант A. Git-based deploy

Подходит для Vercel, Netlify, части конфигураций Timeweb Cloud, Render, Railway.

Схема:

1. Изменения попадают в GitHub.
2. Push в нужную ветку запускает автодеплой.
3. Платформа собирает проект сама.

### Вариант B. CLI deploy

Подходит для Vercel CLI, Railway CLI, Render CLI, Docker CLI и аналогов.

Схема:

1. Агент запускает команду платформы.
2. Получает статус сборки.
3. Проверяет URL релиза.

### Вариант C. SSH / server deploy

Подходит для VPS, Docker-серверов, кастомных Timeweb-конфигураций.

Схема:

1. Подключение к серверу.
2. Обновление кода.
3. Сборка или перезапуск контейнера.
4. Проверка логов и health-check.

## Что обязательно фиксировать после каждого деплоя

Записать в `CHANGELOG.md`, `HANDOFF.md` или release log:

- дата и время;
- среда: staging / production;
- версия или commit;
- что изменилось;
- результат smoke-теста;
- были ли ошибки;
- был ли откат.

## Универсальные вопросы, которые агент должен задать перед production

- Подтверждён ли staging?
- Нужны ли миграции базы данных?
- Нужен ли maintenance window?
- Есть ли риск для действующих пользователей?
- Есть ли команда отката?

- `docs/deploy/ENV-SETUP.md` — переменные и токены простым языком.
- `docs/deploy/MCP-SETUP.md` — вход к настройке MCP.
- `docs/deploy/DEPLOY-MCP-UNIVERSAL.md` — универсальный `mcp.json` для Cursor и других сред.
- `docs/deploy/DEPLOY-TIMEWEB.md` — пошаговый деплой на Timeweb App Platform.
- `docs/deploy/DEPLOY-CHECKLIST.md`, `RELEASE-NOTES.md`, `RUNBOOK.md` — артефакты **конкретного продукта** (spec-driven); общий шаблон чеклиста остаётся в `docs/DEPLOY-CHECKLIST.md`.

## Правило при конфликте инструкций

Если платформа требует особого сценария, то:

1. Сначала читать этот файл.
2. Потом читать платформо-специфичный файл, например `docs/platforms/timeweb-cloud.md` *(файл для Vercel — в разработке)*.
3. Если есть конфликт — побеждает платформо-специфичный файл.
