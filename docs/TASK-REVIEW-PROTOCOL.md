# TASK-REVIEW-PROTOCOL — Протокол проверки задачи

> Этот файл агент читает когда считает задачу выполненной.
> Задача не закрывается без прохождения этого протокола.
> Цель: не оставлять «почти готово» как «готово».

---

## Когда запускается

- Агент написал весь код по шагам задачи
- Агент собирается написать «готово» или «задача выполнена»
- Владелец пишет «проверь» или «всё ли сделано»

---

## Чеклист (агент проходит вслух)

### Блок 1 — Соответствие задаче

- [ ] Цель из раздела «Цель» задачи достигнута?
- [ ] Все шаги из раздела «Шаги» выполнены?
- [ ] Все критерии готовности отмечены `[x]`?

### Блок 2 — Чистота работы

- [ ] Изменения касаются только файлов, связанных с этой задачей?
- [ ] Нет случайно изменённых файлов вне scope?
- [ ] Нет закомментированного мусорного кода?
- [ ] Нет `console.log`, `print`, `TODO` оставленных в коде?

### Блок 3 — Стабильность

- [ ] Проект запускается без ошибок после изменений?
- [ ] Функционал из предыдущих задач не сломан?
- [ ] Если были баги в процессе — они зафиксированы в заметках задачи?

### Блок 4 — Передача контекста

- [ ] `HANDOFF.md` обновлён?
- [ ] Если приняты важные решения — они зафиксированы в заметках задачи?
- [ ] Следующая задача по `ROADMAP.md` определена?

---

## Формат отчёта агента

После прохождения чеклиста агент пишет владельцу:
✅ TASK-XXX завершена

Что сделано: [1-2 предложения]
Что изменилось в проекте: [коротко]
Следующий шаг по ROADMAP: TASK-XXX — [название]

Проблемы/отступления: [если были — описать, если нет — «нет»]

text

## Правило повторных попыток (RETRY RULE)

Если задача не прошла чеклист — агент не бросает и не просит переписать всё заново.

**Алгоритм:**

```
Попытка 1 — исправить конкретный незакрытый пункт
       ↓
Пункт закрыт? → Продолжить чеклист
       ↓
Нет → Попытка 2 — другой подход к тому же пункту
       ↓
Снова неудача? → Стоп. Перейти в MODE = PLANNING
```

**Когда две попытки неудачны, агент пишет:**

> «⚠️ Две попытки закрыть [пункт] не дали результата.
> Возвращаемся в PLANNING — нужно пересобрать подход.
> Вот что не получилось: [описание].
> Предлагаю: [альтернативное решение].»

**Что фиксируется в заметках задачи:**
- Что пытались сделать
- Почему не получилось
- Какое решение принято вместо

> ⚠️ Бесконечный retry — признак неправильно сформулированной задачи,
> а не технической проблемы.

> ⚠️ RETRY RULE — для отдельного пункта чеклиста.
> Если VALIDATION.md вернул **FAIL** целиком → MODE = PLANNING (не retry).

---

## Связь с VALIDATION.md

После прохождения всего чеклиста агент запускает `docs/VALIDATION.md`:
- Чеклист = проверка кода, документации, безопасности (субъективно)
- VALIDATION = объективные тесты «работает / не работает»
- Задача закрывается только если оба пройдены

## Что делать если чеклист не проходит
## Если чеклист не пройден

Если хотя бы один пункт не выполнен — агент:

1. Называет конкретный незакрытый пункт
2. Предлагает что нужно сделать
3. **Не закрывает задачу** до исправления

> ⚠️ «Почти готово» — это не готово.

---

## Особые случаи

| Ситуация | Действие |
|----------|----------|
| Задача оказалась больше чем ожидалось | Разбить на TASK-XXXa и TASK-XXXb |
| Найден баг в чужой задаче | Зафиксировать в заметках, создать отдельную задачу |
| Владелец говорит «и так сойдёт» | Агент фиксирует в HANDOFF.md что пункт пропущен осознанно |
