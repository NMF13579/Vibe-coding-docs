# UX-CHECKLIST-LEGEND.md

Легенда и правила использования UX-чек-листов и режимов.

## 1. Обозначения в чек-листах

Во всех UX-чек-листах используются одинаковые отметки:

- `[x]` — включено по умолчанию  
  Элемент считается стандартом для данного чек-листа.  
  Агент может использовать его без дополнительного одобрения, если соответствующий чек-лист активен.

- `[!]` — обязательно  
  Критичный элемент, без которого сценарий считается небезопасным или некорректным.  
  Агент обязан его учесть, если чек-лист активен (особенно в медицинских и высокорискованных сценариях).

- `[ ]` — опционально  
  Дополнительный элемент. Агент может предложить его, но не обязан включать по умолчанию.  
  Включение зависит от контекста проекта и согласования с пользователем.

## 2. UX-режимы вместо ручного выбора чек-листов

Пользователь **не обязан** выбирать отдельные чек-листы.  
Вместо этого используются режимы из `UX-PRESET-MODES.md`:

- Базовый режим — стандартный web-интерфейс.
- Минимальный режим — упрощённый интерфейс/MVP.
- Медицинский режим — медицинские и чувствительные сценарии.
- Mobile-first режим — мобильный приоритет.
- Data-heavy режим — списки, реестры, таблицы, фильтры.
- Dashboard режим — обзорные панели и KPI.

Каждый режим “под капотом” включает набор чек-листов.  
Агент сам подключает нужные чек-листы в зависимости от выбранного режима.

## 3. Как агент действует в начале проекта

В начале новой проектной беседы (или модуля):

1. Агент по нескольким простым вопросам определяет:
   - тип проекта;
   - наличие медицинских/чувствительных данных;
   - mobile-first или нет;
   - есть ли большие списки, таблицы, дашборды;
   - есть ли разные роли пользователей.

2. Агент выбирает один или несколько режимов из `UX-PRESET-MODES.md`.

3. Агент формулирует предложение пользователю **в терминах режима**, а не чек-листов, например:
   - “Рекомендую базовый UX-режим: он уже включает структуру экранов, формы, ошибки, empty states и доступность.”
   - “Так как проект медицинский, предлагаю медицинский UX-режим с историей изменений и безопасными сценариями.”

4. Пользователь может:
   - подтвердить режим как есть;
   - попросить минимальный режим;
   - попросить добавить/убрать mobile-first, dashboard, data-heavy и т.п.

После подтверждения режима агент считает соответствующие чек-листы активными.

## 4. Как агент использует чек-листы после выбора режима

Когда режим выбран:

- Все элементы с `[!]` из активных чек-листов  
  — обязательны к учёту в проекте.

- Все элементы с `[x]`  
  — используются как стандартный набор без дополнительного согласования.

- Элементы с `[ ]`  
  — агент предлагает как улучшения, когда это уместно, особенно в сложных сценариях.

Агент **не обязан** каждый раз перечислять чек-листы пользователю — ему достаточно следовать им при проектировании интерфейса.

## 5. Если режим не был подтверждён

Если в начале работы UX-режим не выбран или пользователь не ответил:

- Агент использует правила из `UX-CHECKLIST-FALLBACK-RULES.md`.
- Перед созданием **каждого нового раздела** агент:
  - предлагает краткий список подходящих UX-элементов (заголовок, empty states, ошибки, таблица, поиск, история и т.п.);
  - для каждого элемента поясняет:
    - для чего он;
    - является ли он стандартом индустрии или опцией;
  - просит подтвердить, что включать.

Это fallback-режим “по заботе” — когда глобальный UX-набор не зафиксирован на старте, но локально про UX всё равно не забывают.

## 6. Приоритеты выбора

Если возникает конфликт:

- Между “красотой” и доступностью  
  → приоритет у доступности.

- Между “минимализмом” и безопасностью (особенно в медицине)  
  → приоритет у безопасности и предсказуемости.

- Между скоростью внедрения и ясностью интерфейса  
  → приоритет у ясности для пользователя.

## 7. Что важно для медицинских проектов

Для медицинских и около-медицинских проектов агент должен:

- Использовать **медицинский режим** как основной.
- Обязательно активировать чек-листы:
  - `UX-CHECKLIST-MEDICAL.md`
  - `UX-CHECKLIST-ACCESSIBILITY.md`
  - `UX-CHECKLIST-EMPTY-STATES.md`
  - `UX-CHECKLIST-NOTIFICATIONS.md`
  - `UX-CHECKLIST-HISTORY-AND-AUDIT.md`
  - `UX-CHECKLIST-ROLE-BASED-UX.md`

- При необходимости добавить:
  - `UX-CHECKLIST-MOBILE.md` — если есть мобильные сценарии;
  - `UX-CHECKLIST-DASHBOARDS.md` — если есть обзорные панели.

Если пользователь явно не отказывается от медицинского режима, агент обязан ориентироваться на повышенную безопасность и ясность интерфейса.

## 8. Краткое правило для агента

- Сначала выбери **режим**, а не отдельные чек-листы.
- Объясни режим пользователю человеческим языком.
- После согласия следуй чек-листам, входящим в этот режим.
- Если режим не выбран — используй fallback и явно предлагаy UX-элементы при создании разделов.
