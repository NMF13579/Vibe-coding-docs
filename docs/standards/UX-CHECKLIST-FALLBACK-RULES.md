# UX-CHECKLIST-FALLBACK-RULES.md

Правило fallback для агента, если UX-чек-листы не были выбраны в начале проекта.

## 1. Когда применять это правило

Агент применяет это правило, если:
- пользователь не выбрал UX-чек-листы на старте проекта;
- пользователь просит создать новый раздел, страницу, экран, блок или сценарий;
- у агента нет явного указания, какой UX-стандарт использовать.

В этом случае агент не должен сразу собирать интерфейс без согласования UX-основы.

## 2. Что агент делает сначала

Перед созданием нового раздела агент должен предложить пользователю короткий список подходящих UX-элементов для этого экрана.

Для каждого элемента агент должен указать:
- что это такое;
- для какой цели применяется;
- является ли это стандартом индустрии;
- включить ли его в текущий экран по умолчанию или только по запросу.

## 3. Формат предложения

Агент показывает список в понятном виде:

- Название элемента.
- Для чего нужен.
- Когда обычно применяется.
- Это индустриальный стандарт или опция.
- Предлагается ли включить по умолчанию.

Пример формата:

- Заголовок и краткое описание экрана — помогают пользователю сразу понять, где он находится и что делать; это стандарт индустрии; рекомендуется включать по умолчанию.
- Empty state — нужен, чтобы пустой экран не выглядел как поломка; это стандарт индустрии для интерфейсов с данными; рекомендуется включать по умолчанию.
- Поиск — нужен для больших списков и каталогов; это стандарт индустрии только для data-heavy разделов; включать, если элементов много.
- Sticky CTA — нужен, если главное действие должно быть всегда под рукой, особенно на мобильном; это полезная практика, но не обязательный стандарт; включать по ситуации.

## 4. Базовый список элементов, которые агент может предлагать

### 4.1. Почти всегда предлагать

- Заголовок экрана.
- Краткое описание или контекст.
- Один главный CTA.
- Empty state.
- Состояние загрузки.
- Ошибка загрузки / ошибка действия.
- Success state после действия.
- Базовая доступность (контраст, понятные подписи, логичная структура).
- Мобильная адаптация, если экран будет использоваться на мобильных устройствах.

### 4.2. Предлагать по ситуации

- Формы.
- Поиск.
- Фильтры.
- Сортировка.
- Таблица или карточки.
- Подтверждение удаления.
- Уведомления.
- История изменений.
- Дашборд/обзорный блок.
- Помощь/подсказки.
- Role-based различия интерфейса.
- Sticky CTA.
- Bottom sheet / модальное окно.
- Bulk actions.
- Audit trail.

## 5. Как агент определяет, что предлагать

Агент должен смотреть на тип раздела:

### 5.1. Если это форма
Предлагать:
- подписи полей;
- обязательные поля;
- ошибки рядом с полем;
- success state;
- подтверждение опасных действий;
- автосохранение или пошаговую форму, если форма длинная.

### 5.2. Если это список или реестр
Предлагать:
- поиск;
- фильтры;
- сортировку;
- empty state;
- состояние “ничего не найдено”;
- пагинацию или lazy load;
- таблицу или карточки;
- bulk actions, если это реально нужно.

### 5.3. Если это detail page / карточка сущности
Предлагать:
- summary-блок сверху;
- смысловые секции;
- главное действие;
- историю изменений;
- блок связанных данных;
- предупреждения, если действия рискованные.

### 5.4. Если это dashboard
Предлагать:
- KPI/статусы;
- приоритетные списки;
- критичные алерты;
- быстрые действия;
- фильтры периода/статуса;
- мобильную компактную версию, если нужно.

### 5.5. Если это медицинский экран
Предлагать дополнительно:
- подтверждения чувствительных действий;
- безопасные формулировки ошибок;
- role-based интерфейс;
- историю изменений;
- понятные статусы;
- приоритетное отображение критичных данных.

## 6. Как агент помечает элементы

Агент должен использовать такие категории:

- **Стандарт индустрии**  
  Элемент считается нормой для таких экранов и обычно включается по умолчанию.

- **Рекомендуемая практика**  
  Элемент не обязателен всегда, но часто полезен и улучшает UX.

- **Опционально**  
  Элемент нужен только в отдельных сценариях.

## 7. Какой ответ агент должен давать пользователю

Перед созданием экрана агент должен кратко писать, например:

“Перед тем как собирать этот раздел, предлагаю выбрать UX-элементы для него:

- Заголовок + краткое описание — стандарт индустрии, помогает понять экран, предлагаю включить.
- Empty state — стандарт индустрии для экранов с данными, предлагаю включить.
- Поиск — стандарт для больших списков, включить если записей будет много.
- Фильтры — рекомендуемая практика для реестров и журналов, включить если нужно быстро сужать выборку.
- История изменений — рекомендуемая практика, особенно полезна в медицинских и административных сценариях.
- Sticky CTA — опционально, полезно на мобильных экранах.

Подтвердите, какие элементы включаем в этот раздел.”

## 8. Приоритет выбора

Если пользователь не хочет выбирать подробно:
- агент автоматически включает все элементы категории “Стандарт индустрии”;
- предлагает 2–4 элемента категории “Рекомендуемая практика”;
- не навязывает “Опционально”.

## 9. Медицинский приоритет

Если проект медицинский или около-медицинский:
- агент должен особенно предлагать безопасность, понятные статусы, empty states, уведомления, историю изменений, role-based UX и подтверждения чувствительных действий;
- если есть сомнение, агент выбирает ясность и безопасность, а не визуальную сложность.

## 10. Главный принцип

Если UX-чек-лист не выбран заранее, агент сначала предлагает набор подходящих UX-элементов для конкретного раздела, а уже потом проектирует интерфейс.
Это защищает проект от случайного дизайна, уменьшает хаос и делает интерфейс более предсказуемым.
