# UX-CHECKLIST-STARTER-FLOW.md

Правило старта беседы для выбора UX-режима.

## 1. Когда запускать

Агент запускает этот сценарий:
- в начале новой проектной беседы;
- при старте нового модуля;
- если UX-режим ещё не был выбран.

## 2. Цель

Не заставлять пользователя выбирать отдельные UX-чек-листы вручную.
Вместо этого агент сам предлагает один рекомендованный UX-режим.

## 3. Что агент определяет в начале

Агент быстро определяет:

1. Тип проекта:
- обычный кабинет / MVP;
- внутренний сервис / админка;
- медицинский / около-медицинский сервис;
- пациентский кабинет / портал;
- data-heavy система;
- dashboard-ориентированный продукт.

2. Платформу:
- веб;
- mobile-first;
- смешанный сценарий.

3. Уровень сложности:
- минимальный;
- стандартный;
- расширенный.

4. Есть ли:
- чувствительные данные;
- несколько ролей;
- большие списки и фильтры;
- обзорные панели и KPI.

## 4. Что агент делает после этого

Агент выбирает режим из файла `UX-PRESET-MODES.md`.

### 4.1. Базовый режим
Для стандартных проектов.

### 4.2. Минимальный режим
Если нужен простой интерфейс без перегруза.

### 4.3. Медицинский режим
Если проект связан с медициной или чувствительными данными.

### 4.4. Mobile-first режим
Если основной сценарий происходит на мобильном устройстве.

### 4.5. Data-heavy режим
Если в проекте много списков, реестров, таблиц и фильтров.

### 4.6. Dashboard режим
Если важен обзор состояния системы, KPI, приоритетов и очередей.

## 5. Как агент формулирует предложение

Агент должен предлагать не список файлов, а один режим.

Пример 1:
“Для этого проекта рекомендую базовый UX-режим.
Он уже включает нормальную структуру экранов, пустые состояния, ошибки, формы и доступность.
Если хотите, могу упростить его до минимального.”

Пример 2:
“Для этого проекта рекомендую медицинский UX-режим.
Он уже включает безопасные сценарии, понятные статусы, историю изменений, уведомления, role-based UX, доступность и пустые состояния.
При необходимости добавлю mobile-first расширение.”

Пример 3:
“Для этого проекта рекомендую data-heavy режим.
Он уже включает поиск, фильтры, таблицы, пустые состояния и базовую доступность.
Если нужен overview-экран, добавлю dashboard-режим.”

## 6. Что пользователь может ответить

Пользователю не нужно знать имена файлов.
Он может ответить просто:

- “Да, применяем.”
- “Сделай минимальный вариант.”
- “Добавь mobile-first.”
- “Без dashboard.”
- “Оставь только базу.”
- “Добавь медицинские требования.”

## 7. Что делает агент после согласия

После согласия агент:
- фиксирует выбранный режим;
- использует чек-листы, входящие в него;
- строит интерфейс по этим стандартам;
- не заставляет пользователя заново выбирать UX-элементы на каждом шаге.

## 8. Если режим не выбран

Если пользователь не подтвердил режим:
- агент использует `UX-CHECKLIST-FALLBACK-RULES.md`;
- перед созданием нового раздела предлагает краткий список подходящих UX-элементов для конкретного экрана.

## 9. Главный принцип

На старте беседы пользователь должен видеть не сложную систему документов, а понятную рекомендацию:
“Вот подходящий UX-режим для вашего проекта, он уже включает всё основное.”
