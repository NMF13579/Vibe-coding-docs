# WORKFLOW — конвейер задач

---

## Обязательный порядок

1. Получить задачу
2. Прочитать ROADMAP — соответствует ли текущему этапу?
   *(если ROADMAP ещё не создан — пропустить этот шаг; ROADMAP создаётся в рамках START.md)*
3. Если используется Git и это крупная задача — предложить сделать коммит текущего состояния как точку возврата
4. Составить план
5. Показать план владельцу
6. Написать: «Подтвердите выполнение плана.»
7. Дождаться: «да» / «выполняй» / «go»
8. Писать код

> **Молчание ≠ согласие.** Если владелец не ответил на план — один раз мягко переспросить:
> «Жду подтверждения — продолжать по плану выше?»
> Если ответа снова нет — оставаться на паузе. Начинать работу без явного «да» запрещено.

## Перед реализацией экрана

Если задача затрагивает **новый экран** или **заметно меняет** то, что видит пользователь (не одна правка формулировки при неизменной структуре):

1. В `docs/processes/PROCESSES.md` есть шаг процесса, которому экран соответствует (или явно записано, что экран вне процесса — и зачем он тогда нужен).
2. В `docs/ux/SCREEN-MAP.md` есть этот экран (или понятный блок навигации к нему).
3. В `docs/ux/WIREFRAMES.md` есть схема экрана (блоки сверху вниз или отсылка «как у экрана X»).
4. В `docs/ux/` согласована цепочка записей: `PAGES.md` и по смыслу `TEMPLATES.md`, `ORGANISMS.md`, `MOLECULES.md`, `ATOMS.md` — настолько, чтобы было ясно, что на экране и из чего он собран (порядок см. `docs/ux/ATOMIC-DECOMPOSITION.md`).

Если чего-то из перечисленного **нет** — **не** писать код и **не** заводить в `tasks/` задачу на реализацию этого экрана; сначала дописать документацию и согласовать с владельцем **коротким человеческим резюме** (без внутреннего жаргона).

*Исключение:* правка только текста или мелочи без изменения шагов, полей, блоков или сценария — при условии, что след в документах для этого экрана уже был.

## Формат плана

```
## План: [название]

**Цель:** [что должно работать после]
**Этап ROADMAP:** [этап X]
**Затронутые файлы:** [список]

**Шаги:**
1. [действие]
2. [действие]

**Риски:** [что может пойти не так]
**Не трогаю:** [что остаётся без изменений]

Подтвердите выполнение плана.
```

## После выполнения

```
## Выполнено: [название]

**Что сделано:**
- [пункт]

**Изменённые файлы:**
- [файл] — [что изменено]

**Как проверить:**
1. [шаг]
```

Обязательно обновить: HANDOFF.md, memory-bank/project-status.md

## Прерывание плана (откат или экстренная ситуация)

Если в процессе выполнения плана пользователь пишет «откати», «всё сломалось», «стоп»:
1. Немедленно остановить выполнение
2. Зафиксировать в HANDOFF.md: на каком шаге остановились и что было сделано
3. Перейти к `docs/ROLLBACK-PROTOCOL.md`
4. После стабилизации — вернуться к плану с того места, где остановились

> Незавершённый план лучше потерянного контекста.

---

## Когда останавливаться

- Задача затрагивает данные пользователей или auth
- Нужно удалить, переименовать таблицу или поле в БД
- Предлагается новая внешняя зависимость
- Задача выходит за рамки ROADMAP
- Непонятно что хочет владелец

Формула: «Стоп. Прежде чем продолжить — мне нужно уточнить: [вопрос].»
