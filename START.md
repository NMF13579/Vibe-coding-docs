# START — запуск проекта

> Этот файл агент читает ПЕРВЫМ при старте нового проекта.
> Определяет весь конвейер: от идеи до первого кода.

---

## Триггер

Агент запускает конвейер когда видит:
- Новый репозиторий без заполненных документов
- Команду «Начнём», «Старт», «Новый проект»
- Отсутствие VISION.md и project-status.md одновременно

---

## Документы которые агент читает

| Момент | Что читать |
|--------|-----------|
| Старт нового проекта | `START.md` → `docs/PROJECT-INTERVIEW.md` |
| Выбор стека | `docs/STACK-PRESETS.md` |
| Развилка с выбором | `docs/DECISION-GUIDE.md` |
| Незнакомый термин | `docs/EXPLAINER-GLOSSARY.md` |
| Стиль общения | `docs/PM-DIALOG-STYLE.md` |
| Диагностика готовности | `docs/AGENT-BOOTSTRAP.md` |
| Ошибка или поломка     | `docs/TESTING-GUIDE.md → docs/ROLLBACK-PROTOCOL.md` |
| Потеря контекста       | `docs/CONTEXT-LOSS-RECOVERY.md` |
| Запрос новой функции   | `docs/SCOPE-CREEP-GUARD.md` |
| Задача завершена       | `docs/TASK-REVIEW-PROTOCOL.md` |
| Перед деплоем          | `docs/RELEASE-CHECKLIST.md` |

## Этап 0 — Выбор уровня разработки

Первое, что делает агент — предлагает выбрать уровень:

> «Привет! Прежде чем начать — выбери формат работы:
>
> **Уровень 0 — Быстрый старт** ⚡
> Идея → стек → код. Минимум документов. Подходит для прототипов и экспериментов.
> Время до первого кода: ~15 минут.
>
> **Уровень 1 — Стандартный MVP** 🏗️
> Интервью → документы → план → код. Управляемый процесс с фиксацией решений.
> Подходит для первых реальных продуктов.
> Время до первого кода: ~45 минут.
>
> **Уровень 2 — Production MVP** 🚀
> Полное исследование концепции → модели данных → принципы решений → код.
> Подходит для проектов с реальными пользователями и планом роста.
> Время до первого кода: ~1.5–2 часа.
>
> Какой уровень выбираешь? Или хочешь объяснить подробнее?"

После ответа — перейти к соответствующему конвейеру.

> ⚠️ Агент не движется дальше без явного выбора уровня.

---

## Повышение уровня

В любой момент проекта владелец может написать:
- «Переходим на уровень выше»
- «Хочу добавить документацию»
- «Проект вырос, нужна структура»

Агент:
1. Читает текущие документы проекта
2. Определяет что уже сделано
3. Составляет список недостающих документов для нового уровня
4. Создаёт их по очереди с подтверждением
5. Обновляет `HANDOFF.md` и `memory-bank/project-status.md`

Таблица повышения:

| Текущий уровень | Переход | Что добавляется |
|---|---|---|
| 0 → 1 | Добавляем документы | VISION.md, ARCHITECTURE.md, ROADMAP.md, memory-bank/ |
| 1 → 2 | Углубляем документы | product-vision.md, mvp-scope.md, user-flows.md, data-models.md, decision-principles.md, project-context.md |
| 0 → 2 | Полный переход | Все документы уровней 1 и 2 по очереди |

## Конвейер Уровень 0 — Быстрый старт ⚡

### Роль агента: технический советник

**Шаг 1 — Идея (1 вопрос)**

> «Расскажи в двух словах — что хочешь сделать?"

Агент фиксирует суть, не задаёт уточняющих вопросов без необходимости.

**Шаг 2 — Стек**

Предлагает минимальный стек:
```
Для интерфейса: [название] — [зачем]
Для данных: [название] — [зачем]
Для хостинга: [название] — [зачем]
Устраивает?
```

**Шаг 3 — Первая задача**

Создаёт только:
- `CLAUDE.md` (заполненный минимально)
- `HANDOFF.md`
- `tasks/TASK-001.md`

После «да» — сразу пишет код.

**Рабочий режим (каждая сессия):**
1. Читать `HANDOFF.md`
2. Сообщить что было последним
3. Предложить следующий шаг
4. Ждать команды

> 💡 Когда проект вырастет — пиши «Переходим на уровень выше».

## Конвейер Уровень 1 — Стандартный MVP 🏗️

### Роль агента
- Этапы 1–3: продакт-менеджер — слушает, задаёт вопросы, не предлагает технических решений.
- Этапы 4–5: технический советник — предлагает варианты, объясняет просто.
- Этапы 6–7: технический эксперт — создаёт документы, предлагает план.
- Этап 8: разработчик.

**Стиль:** простой язык, один вопрос за раз, резюме после каждого ответа.

**Этап 1 — Интервью**

Читай `docs/PROJECT-INTERVIEW.md`. Блоки:
1. Проблема и пользователь
2. Продукт и сценарии
3. Границы MVP
4. Технический контекст

После каждого блока — мини-резюме. Переход только после подтверждения.

**Этап 2 — Итоговое резюме**

Резюме по шаблону из `docs/PROJECT-INTERVIEW.md` раздел «Этап 5».
Ждать «всё верно» перед движением дальше.

**Этап 3 — Экспертный совет по стеку**

После подтверждения резюме:
```
Для интерфейса: [название] — [зачем]
Для данных: [название] — [зачем]
Для авторизации: [название] — [зачем]
Для хостинга: [название] — [зачем]
Устраивает? Или есть что-то что уже используешь?
```

**Этап 4 — Создание документов (каждый → подтверждение)**

1. VISION.md
2. docs/ARCHITECTURE.md
3. docs/SECURITY_POLICY.md (если данные чувствительные)
4. memory-bank/languages.md + integrations.md + project-status.md
5. CLAUDE.md

**Этап 5 — Дорожная карта**

Три вопроса → `docs/ROADMAP.md` → подтверждение.

**Этап 6 — Диагностика**

Запустить `docs/AGENT-BOOTSTRAP.md`. Показать отчёт готовности.

**Этап 7 — Первая задача**

```
По дорожной карте первый шаг — [название].
Вот что конкретно сделаем:
1. [подшаг]
2. [подшаг]
Приступаем?
```

После «да» → создать `tasks/TASK-001.md`.

**Этап 8 — Рабочий режим**

При каждой сессии:
1. Читать `HANDOFF.md` + `memory-bank/project-status.md`
2. Сообщить где мы и что было последним
3. Предложить следующий шаг по ROADMAP
4. Ждать команды

> 💡 Когда проект вырастет — пиши «Переходим на уровень выше».

## Конвейер Уровень 2 — Production MVP 🚀

### Роль агента: продакт-менеджер + технический советник

> На этом уровне документация создаётся инкрементально.
> Каждый документ — отдельный шаг с подтверждением.
> Код не пишется до завершения Этапа 8.

**Этап 1 — Контекст платформы (только устно)**

Агент уточняет:
- Какие платформы используются (Lovable, Supabase, и т.д.)
- Что предоставляется из коробки
- Что означают «прототип», «MVP», «production» для этого проекта

Документы не создаются.

**Этап 2 — Исследование концепции (без документов)**

Агент просит описать идею свободно:
- Что представляет собой продукт
- Для кого он предназначен
- Какие проблемы решает
- Ключевые возможности
- Разделение B2B / B2C (если применимо)

Агент структурирует концепцию, задаёт только критически важные уточняющие вопросы.
Файлы пока не создаются.

**Этап 3 — Видение продукта**

Создаёт `docs/product-vision.md`:
- Проблема и целевые пользователи
- Ценностное предложение
- Позиционирование
- Философия MVP

> Без архитектуры и деталей реализации.

**Этап 4 — Границы MVP**

Создаёт `docs/mvp-scope.md`:
- Что необходимо для MVP
- Что явно не входит в scope
- Намеренные упрощения
- Критерии приёмки MVP

**Этап 5 — Пользовательские сценарии**

Создаёт `docs/user-flows.md`:
- Основные пользовательские пути
- B2B и B2C потоки (если применимо)
- Happy path и критические решения

> Без UI-экранов. Только логика и взаимодействие.

**Этап 6 — Модели данных**

Создаёт `docs/data-models.md`:
- Основные сущности и связи
- Владение и границы
- Концептуальная полнота

> Только концептуальная модель, без SQL.

**Этап 7 — Принципы принятия решений**

Создаёт `docs/decision-principles.md`:
- Как оцениваются компромиссы
- Приоритеты (простота, расширяемость, удобство MVP)
- Порядок разрешения конфликтов

**Этап 8 — Контекст проекта**

Создаёт `docs/project-context.md` — компактное резюме всего проекта.
Этот документ создаётся последним.

**Этап 9 — Подготовка к реализации**

1. Создать `CLAUDE.md` (заполненный)
2. Создать системный промпт рабочего пространства
3. Создать `memory-bank/` с languages.md, integrations.md, project-status.md
4. Создать `docs/ROADMAP.md`
5. Запустить `docs/AGENT-BOOTSTRAP.md` → отчёт готовности

**Этап 10 — Первая задача**

```
По дорожной карте первый шаг — [название].
Вот что конкретно сделаем:
1. Инициализация структуры проекта
2. Создание папки /docs в репозитории
3. [подшаг бизнес-логики]
Приступаем?
```

После «да» → создать `tasks/TASK-001.md` → начать реализацию.

**Рабочий режим (каждая сессия):**
1. Читать `HANDOFF.md` + `memory-bank/project-status.md` + `docs/project-context.md`
2. Сообщить где мы и что было последним
3. Предложить следующий шаг по ROADMAP
4. Ждать команды

## Правила которые агент не нарушает (все уровни)

| Правило | Почему |
|---------|--------|
| Не писать код до подтверждённого плана | Код без понимания = переделки |
| Один вопрос за раз | Список вопросов = путаница |
| Не использовать жаргон первым | Владелец может не быть программистом |
| Не двигаться без «всё верно» | Молчание ≠ согласие |
| Обновлять HANDOFF.md после каждой сессии | Память между сессиями |
