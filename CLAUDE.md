# CLAUDE.md — главные инструкции для Claude Code

Этот репозиторий — **docs-first** шаблон. Для **Claude Code / Claude Projects** главный поток такой:

- **Живое ТЗ продукта:** `project/PROJECT.md` (единый источник истины по продукту в этом потоке).
- **Этапы и узкий контекст:** `stages/0X-*/BOOT.md` — что читать, куда писать, smoke-test.
- **Общие правила на все этапы:** `shared/` (кратко; полнота — в `LAYER-1/` … `LAYER-3/`).

Другие среды (Cursor, Copilot и т.д.) не являются источником истины для **этой** схемы; им можно отдавать тот же `CLAUDE.md` как эталон поведения.

---

## Роль

- Помогать владельцу (часто не-разработчику) вести продукт от идеи до MVP и дальше.
- Простой язык; **один вопрос за раз** на развилках.
- Не перегружать терминами; не вводить жаргон первым.

---

## Обязательный цикл: Plan → Review → Confirm → Execute

1. Сформулируй **короткий план на русском** (что сделаешь, какие файлы тронешь).
2. Покажи владельцу и **дождись явного подтверждения** (да / ок / выполняй).
3. Только после этого вноси правки в файлы или пиши код.
4. Если результат неудачен — вернись к плану, не «латай» вслепую.

**Молчание ≠ согласие.** Без подтверждения плана работу не начинать.

---

## Правило задачи

**Один запрос = одна задача.** Крупное дроби; расширения скоупа — отдельное согласование (`shared/scope-guard.md`).

---

## Явные пробелы вместо догадок

Если данных не хватает — пиши в тексте:

`❓ НЕ ОПРЕДЕЛЕНО — что нужно уточнить`

Не заполняй пробелы вымыслом.

---

## Закрытые секции в `project/PROJECT.md`

У каждой секции в комментариях есть `LOCKED:true` или `LOCKED:false`.

- С **`LOCKED:true`** не редактировать без **явного** решения владельца.
- Этап **не переписывает** чужие секции: Interview не лезет в Deploy; Dev не переписывает Discovery при lock.

---

## Автозапуск сессии (ОБЯЗАТЕЛЬНО)

При **первом** сообщении в новой сессии — автоматически:

1. Прочитай **`project/PROJECT.md`** (хотя бы YAML и заголовки секций) — где продукт по единому ТЗ.
2. Прочитай **`HANDOFF.md`** и **`LAYER-3/project-status.md`** — состояние **шаблона** репозитория и последняя работа.
3. Прочитай **`LAYER-3/lessons.md`** и **`LAYER-3/PROJECT-MEMORY.md`**, если есть.
4. Прочитай и применяй правила самопроверки: **`LAYER-1/self-verification.md`** (перед ответами с кодом — по триггерам из файла; пользователю не напоминать).
5. Если в `HANDOFF.md` одна и та же задача незакрыта **второй раз подряд** — предупреди:
   «⚠️ Задача [название] идёт уже несколько сессий. Разбить на части или продолжить?»
6. Кратко (3–5 строк): где мы; что было в прошлый раз; 1–3 варианта следующего шага.

**Важно:** шаги 1–6 — контекст для владельца. Пока нет выбранного шага и **подтверждённого** плана — код не писать.

**Если `HANDOFF.md` нет** — проект не инициализирован как шаблон: веди по `llms.txt` и `LAYER-1/workflow.md` (Этап 0) или начни с `stages/01-interview/BOOT.md` и пустого `project/PROJECT.md`.

## Маршрут по этапам (узкий контекст)

Владелец задаёт этап, читая соответствующий файл:

| Этап | Старт |
|------|--------|
| Интервью / Discovery | `stages/01-interview/BOOT.md` |
| UX | `stages/02-ux/BOOT.md` |
| Разработка / Specs | `stages/03-dev/BOOT.md` |
| Деплой | `stages/04-deploy/BOOT.md` |

В каждом `BOOT.md`: что читать, куда писать в `PROJECT.md`, приоритет правил, **smoke-test** перед работой.

Обзор системы для человека: [`CLAUDE-CODE-FLOW.md`](./CLAUDE-CODE-FLOW.md). Перед релизом: `CHECKLIST.md`.

---

## Smoke-test (перед изменениями по этапу)

Когда работаешь по `BOOT.md`, **в первом ответе после чтения** явно назови:

1. Среду (**Claude Code** или какая открыта).
2. Номер и имя этапа из `BOOT.md`.
3. Что **нельзя** делать (чужие секции, `LOCKED:true`, скоуп).
4. **Куда** пишешь результат в `project/PROJECT.md`.

Если файл или секция не найдены — скажи явно.

---

## Автосохранение контекста шаблона (ОБЯЗАТЕЛЬНО)

После завершения **задачи**, затрагивающей репозиторий-шаблон:

1. Обнови **`HANDOFF.md`** — что сделали, где остановились, что дальше.
2. Обнови **`LAYER-3/project-status.md`**.
3. По таблице ниже — при необходимости остальные `LAYER-3/*`:

| Событие | Файл |
|--------|------|
| Баг | `LAYER-3/fixes.md` |
| Фича | `LAYER-3/features.md` |
| Урок | `LAYER-3/lessons.md` |
| Интеграция | `LAYER-3/integrations.md` |
| Стек / языки | `LAYER-3/languages.md` |
| Безопасность | `LAYER-3/security.md` |
| Отложено | `LAYER-3/deferred-decisions.md` |
| Закрыта задача | `LAYER-3/PROJECT-MEMORY.md` |

4. Сообщи: «Контекст сохранён ✅»

Напоминание о сохранении: **10+ правок** файлов или **план из 3+ шагов** без обновления `HANDOFF.md` → «⚠️ Контекст не сохранялся. Сохранить сейчас?»

**Продуктовые решения** по возможности дублируй в **`project/PROJECT.md` → DECISIONS** (дата, решение, почему). Версию в YAML обновляй после значимых изменений (`version`, `updated`, `stage`).

## Что читать первым (по ситуации)

1. `README.md` — обзор шаблона.

3. **`stages/*/BOOT.md`** — текущий этап (по запросу владельца).
4. `llms.txt` — маршруты по темам к `LAYER-1/` / `LAYER-2/`.

## Приоритет источников при конфликте

1. Документ задачи в `tasks/`, если есть.
2. **`project/PROJECT.md`** (продукт) + **`HANDOFF.md`** + **`LAYER-3/project-status.md`** (шаблон).
3. Активный **`stages/*/BOOT.md`** (frontmatter `priority`).
4. `shared/scope-guard.md` → `shared/agent-contract.md` → локальный документ этапа → `shared/workflow.md`.
5. Профильный файл в `LAYER-1/` или `LAYER-2/` для темы.
6. Этот файл и `LAYER-1/system-prompt.md`.

## Контроль интервью

Если ведёшь интервью по `LAYER-2/discovery/project-interview.md`: после каждого шага — self-check из `LAYER-1/tools/adapters/CLAUDE-INTERVIEW-CONTROL.md` (✅/⚠️/❌). При **❌** не переходи к следующему вопросу, пока не исправлено.

Итог фиксируй в **`project/PROJECT.md` → DISCOVERY** согласно `stages/01-interview/BOOT.md`.
