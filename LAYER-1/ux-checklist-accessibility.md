# UX-CHECKLIST-ACCESSIBILITY.md

# UX-CHECKLIST-ACCESSIBILITY.md

Чек-лист доступности (a11y) для агента.
Применяется ко всем интерфейсам, особенно для медицинских и мобильных.
`[x]` — включено по умолчанию, `[!]` — обязательно, `[ ]` — опционально.

## 1. Общие принципы

- [!] Основные сценарии должны быть доступны без мыши.
- [!] Смысл не должен передаваться только цветом.
- [!] Текст должен быть читаем без зума и танцев с бубном.
- [x] Критичные действия и статусы должны быть понятны любому пользователю.
- [ ] Отдельный обзор доступности перед релизом.

## 2. Текст и контраст

- [!] Контраст текста и фона соответствует WCAG AA.
- [!] Основной текст не мельче базового размера (обычно 16 px).
- [x] Второстепенный текст остаётся читаемым.
- [x] Текст на цветных кнопках/бейджах читабелен.
- [x] Нет серого текста на слегка более сером фоне.
- [ ] Проверка контраста для всех ключевых элементов отдельно.

## 3. Цвет и смысл

- [!] Цвет не единственный способ обозначить статус.
- [!] Ошибки и предупреждения имеют и текст, и цвет.
- [x] Важные статусы дополнительно маркируются иконкой или текстом.
- [x] Легенды к цветам в графиках и таблицах понятны.
- [ ] Добавлены текстовые метки к цветовым индикаторам.

## 4. Клавиатурная навигация

- [!] Все основные действия доступны с клавиатуры.
- [!] Фокус по Tab двигается логично и предсказуемо.
- [!] Фокус видно (подсветка, рамка, смена состояния).
- [x] Кнопки и ссылки можно активировать Enter/Space.
- [ ] Есть "перейти к основному контенту" (skip link).
- [ ] Комбинации клавиш для частых действий (где уместно).

## 5. Screen reader и семантика

- [!] Заголовки выстроены по уровню (h1, h2, h3 без хаоса).
- [!] Основные блоки размечены семантически (header, main, nav, footer).
- [!] Интерактивные элементы имеют корректные роли (button, link и т.п.).
- [x] У иконок‑кнопок есть понятный текст (aria-label).
- [x] У картинок есть alt или явное пустое alt, если картинка декоративная.
- [ ] ARIA-атрибуты используются только там, где действительно нужны.

## 6. Формы и ошибки

- [!] Каждое поле имеет подпись (label).
- [!] Ошибка привязана к конкретному полю.
- [!] Ошибки объясняют, что сделать, а не только “неверное значение”.
- [x] Обязательные поля явно отмечены.
- [x] Сообщения об успехе и ошибке понятны на слух.
- [ ] Есть общее summary ошибок для длинных форм.

## 7. Фокус, модальные окна, диалоги

- [!] При открытии модального окна фокус переходит внутрь него.
- [!] При закрытии модального окна фокус возвращается туда, откуда пришёл.
- [!] В модальных окнах есть явная кнопка закрытия.
- [x] Escape закрывает диалог, если это безопасно.
- [x] Пользователь не может “потерять” фокус в невидимой области.
- [ ] Тёмная подложка под модальным окном не мешает работе screen reader.

## 8. Мобильная доступность

- [!] Интерактивные элементы не менее 44x44 px.
- [!] Расстояние между touch‑целями достаточное.
- [!] Основные действия доступны без точного прицеливания.
- [x] Интерфейс корректно работает в портретной ориентации.
- [ ] Поддерживается landscape, если сценарий этого требует.
- [ ] Учитываются системные жесты (без конфликтов).

## 9. Состояния элементов

- [!] Есть состояния: обычное, наведение (если есть), фокус, нажатие, отключено.
- [!] Состояния понятны не только по цвету (форма, иконка, текст).
- [x] Disabled выглядит реально недоступным, но читабельным.
- [x] Загрузка отображается явно (спиннер, skeleton, текст).
- [ ] Важные долгие операции имеют индикатор прогресса.

## 10. Контент и язык

- [!] Формулировки максимально ясные, без лишнего жаргона.
- [!] Длинные фразы разбиты на удобочитаемые блоки.
- [x] Сокращения и аббревиатуры объясняются там, где это критично.
- [x] Важное не прячется в “мелкий серый текст”.
- [ ] Есть глоссарий, если много спецтерминов.

## 11. Медицинский контекст

- [!] Сообщения не вызывают лишней тревоги, если ситуация не критична.
- [!] Важные статусы (например, критические результаты) обозначены ясно.
- [!] Ошибки не порождают ощущение “данные пациента потеряны”.
- [x] Подробные технические детали скрыты за дополнительным действием.
- [ ] Разные роли (врач, пациент, администратор) получают свой уровень детализации.

## 12. Проверка и поддержка

- [x] Есть базовый чек‑лист, по которому агент проходит перед релизом.
- [x] Основные сценарии проверены с клавиатурой.
- [x] Основные сценарии проверены на мобильном устройстве.
- [ ] Проведено отдельное тестирование доступности для медицинских интерфейсов.
- [ ] Запланирован регулярный пересмотр доступности по мере развития продукта.

## 13. Правило применения

- Агент использует все пункты `[!]` как обязательные для любого интерфейса.
- Пункты `[x]` считаются стандартом доступности по умолчанию.
- Пункты `[ ]` агент предлагает как дополнительные меры для повышенной доступности.
- В случае конфликта между визуальным “вау” и доступностью агент выбирает доступность и понятность.

## 14. Фраза для согласования с пользователем

"Могу включить стандартный чек‑лист доступности: контрастный текст, понятные заголовки, работа с клавиатуры, корректные сообщения об ошибках, доступные формы и мобильные touch‑элементы. Обязательные пункты будут учтены во всех экранах. Применяем?"

Пункты `[x]` включаются по умолчанию.

## 1. Общий принцип

- [!] Один главный сценарий на экран.
- [!] Интерфейс должен быть удобен для использования одной рукой.
- [!] Ключевые действия должны быть доступны без точного прицеливания.
- [!] Экран не должен быть перегружен.
- [!] Пользователь должен понимать, что делать дальше, без лишнего чтения.

## 2. Структура экрана

- [x] Заголовок экрана.
- [x] Короткое описание или контекст экрана.
- [!] Один главный CTA.
- [x] Вторичные действия визуально слабее главного.
- [x] Главная информация расположена выше второстепенной.
- [ ] Блок подсказок.
- [ ] Блок “что делать дальше”.

## 3. Навигация

- [!] Простая и понятная мобильная навигация.
- [!] Критичные действия находятся в зоне удобного нажатия.
- [x] Текущий раздел выделен явно.
- [x] Возврат назад понятен и предсказуем.
- [ ] Bottom navigation bar.
- [ ] Sticky bottom actions.
- [ ] Быстрый доступ к частым действиям.

- [!] Размер интерактивных элементов не меньше 44x44 px.
- [!] Между важными touch-элементами достаточно расстояния.
- [!] Кнопки и ссылки легко нажимаются пальцем.
- [!] Для важных действий есть альтернатива swipe-only жестам.
- [x] Нажатие даёт визуальную обратную связь.
- [ ] Поддержка haptic feedback.
- [ ] Gesture shortcuts для второстепенных действий.

## 5. Формы

- [!] Поля ввода удобны на мобильном экране.
- [!] Поля и кнопки имеют достаточную высоту.
- [!] Подписи полей понятны без жаргона.
- [!] Ошибки показываются рядом с полем.
- [x] Логичный порядок полей сверху вниз.
- [x] Главная кнопка формы заметна.
- [ ] Деление длинной формы на шаги.
- [ ] Автосохранение черновика.
- [ ] Подсказки под сложными полями.
- [ ] Автоподстановка и маски для телефонов, дат и кодов.

## 6. Состояния экранов

- [x] Состояние загрузки.
- [x] Пустое состояние.
- [x] Ошибка загрузки.
- [x] Состояние "ничего не найдено".
- [x] Состояние успешного действия.
- [ ] Skeleton loader.
- [ ] Empty state для первого входа.
- [ ] Pull-to-refresh для подходящих экранов.

## 7. Empty states

- [x] Понятно объяснено, почему экран пуст.
- [x] Есть следующий шаг.
- [x] Есть кнопка первого действия.
- [ ] Иллюстрация или иконка.
- [ ] Demo content.
- [ ] Специальный empty state для первого использования.

## 8. Списки и карточки

- [x] Основное действие по элементу понятно сразу.
- [x] Список легко сканируется взглядом.
- [x] Главные данные видны без открытия карточки.
- [ ] Замена сложных таблиц на карточки.
- [ ] Sticky filters.
- [ ] Быстрые действия по элементу.
- [ ] Группировка элементов по секциям.

## 9. Кнопки и действия

- [x] Primary button для главного действия.
- [x] Secondary button для вторичных действий.
- [x] Danger style для удаления.
- [x] Disabled state для недоступных действий.
- [!] Подписи кнопок ясные и короткие.
- [ ] Иконки в кнопках.
- [ ] Sticky CTA внизу экрана.
- [ ] Floating action button.

## 10. Модальные окна и bottom sheets

- [x] Подтверждение удаления.
- [x] Подтверждение выхода без сохранения.
- [x] Понятный заголовок модального окна.
- [!] Модальные окна не должны ломать основной сценарий.
- [ ] Bottom sheet вместо большого modal.
- [ ] Дополнительный текст с последствиями действия.

## 11. Обратная связь и ошибки

- [x] Сообщение об успехе после действия.
- [x] Понятное сообщение об ошибке.
- [x] Ошибка объясняет, что делать дальше.
- [x] Ошибки формы показываются рядом с полем.
- [ ] Toast для второстепенных событий.
- [ ] Retry button.
- [ ] Inline alert в верхней части экрана.

## 12. Доступность

- [!] Контрастный текст.
- [!] Смысл не передаётся только цветом.
- [!] Интерфейс работает в portrait и landscape, если это требуется сценарием.
- [!] Поддержаны разные способы ввода, не только касание.
- [!] Видимый фокус и понятный порядок перехода.
- [x] Понятные подписи кнопок и полей.
- [ ] Поддержка screen reader-сценариев.
- [ ] Расширенная accessibility-проверка.

## 13. Читаемость и когнитивная нагрузка

- [!] Минимум лишнего текста на экране.
- [!] Важная информация выше второстепенной.
- [!] Один экран не должен пытаться решить слишком много задач.
- [x] Короткие и понятные подписи.
- [ ] Пошаговые сценарии для сложных процессов.
- [ ] Режим повышенной читаемости.

## 14. Правило применения

- Агент использует все пункты `[!]` как обязательные для мобильного интерфейса.
- Агент использует все пункты `[x]` как стандартный mobile UX-набор.
- Пункты `[ ]` предлагает как дополнительные улучшения.
- Если есть конфликт между плотностью интерфейса и удобством нажатия, агент выбирает удобство нажатия.
- Если экран перегружен, агент упрощает его до одного главного сценария.

## 15. Фраза для согласования с пользователем

"Могу собрать мобильный интерфейс по стандартному mobile UX-чек-листу: один главный сценарий на экран, удобные touch-зоны, простая навигация, понятные формы, состояния загрузки и ошибок, и доступный mobile-first интерфейс. Необязательные элементы можно отключить. Применяем?"

# UX-CHECKLIST-EMPTY-STATES.md

# UX-CHECKLIST-EMPTY-STATES.md

UX-чек-лист пустых состояний для агента.
Используется для экранов без данных, после фильтра/поиска, при первом входе и при ошибках загрузки.
Пункты `[x]` включаются по умолчанию.
Пункты `[ ]` можно добавить по запросу.

## 1. Общие принципы

- [x] Пустой экран не должен выглядеть как ошибка системы.
- [x] Пользователь должен понимать, что происходит и что делать дальше.
- [x] Empty state — это возможность объяснить, как пользоваться интерфейсом.
- [x] Текст должен быть коротким и понятным.
- [ ] Отдельные empty states для разных контекстов (первый вход, фильтр, ошибка).

## 2. Базовый empty state (нет данных вообще)

- [x] Короткое пояснение, почему здесь пусто.
- [x] Подсказка, что нужно сделать, чтобы появились данные.
- [x] Одна основная кнопка первого действия.
- [ ] Дополнительная ссылка на документацию или помощь.
- [ ] Краткий пример, как выглядит заполненный экран.
- [ ] Ненавязчивая иллюстрация или иконка.

## 3. Empty state после фильтра/поиска

- [x] Сообщение "ничего не найдено".
- [x] Подсказка, что можно изменить фильтр или поиск.
- [x] Кнопка "сбросить фильтры" или аналогичное действие.
- [ ] Предложение часто используемых фильтров или запросов.
- [ ] Примеры запросов, которые дадут результат.
- [ ] Индикатор активных фильтров, который легко сбросить.

## 4. Empty state после удаления данных

- [x] Сообщение, что элементов больше нет.
- [x] Понятный контекст: что именно было удалено.
- [x] Подсказка, можно ли и как создать новый элемент.
- [ ] Временное уведомление о том, что данные удалены.
- [ ] Ссылка на историю изменений или лог, если он есть.

## 5. Empty state при первом использовании (onboarding)

- [x] Сообщение, что это первый вход или пустой старт.
- [x] Подсказка, как начать работать.
- [x] Одна главная кнопка “начать” / “создать первый элемент”.
- [ ] Короткая инструкция шаг за шагом (1–2 шага).
- [ ] Ссылка на подробную инструкцию или обучение.
- [ ] Пример заполненного состояния для ориентира.

## 6. Empty state из-за отсутствия прав доступа

- [x] Сообщение, что у пользователя нет доступа к этой информации.
- [x] Пояснение без технического жаргона.
- [x] Подсказка, к кому обратиться для получения доступа.
- [ ] Кнопка "запросить доступ" или аналогичное действие.
- [ ] Ссылка на политику доступа или роли.

## 7. Empty state при ошибке загрузки (состояние после ошибки, а не только тост)

- [x] Сообщение, что произошла ошибка при загрузке данных.
- [x] Пояснение, что это может быть временная проблема.
- [x] Кнопка “повторить попытку”.
- [ ] Короткий технический текст для поддержки (код ошибки, если уместно).
- [ ] Ссылка на статус сервиса или контакт поддержки.

## 8. Тон и текст

- [x] Текст короткий и конкретный.
- [x] Без обвинения пользователя.
- [x] Без агрессивного юмора или сарказма.
- [x] Формулировки вежливые и спокойные.
- [ ] Адаптация тона под аудиторию (пациент, врач, администратор, внутренний пользователь).

## 9. Визуальное оформление

- [x] Пустое состояние визуально отличается от “сломавшегося” экрана.
- [x] Основной фокус на тексте и кнопке действия.
- [x] Нет лишнего визуального шума.
- [ ] Ненавязчивая иллюстрация/иконка, поддерживающая смысл.
- [ ] Одинаковый визуальный стиль empty states по всему продукту.

## 10. Медицинский контекст (если проект медицинский)

- [x] Empty state не должен создавать ощущение, что “данные пациента исчезли”.
- [x] Явно указано, что данных ещё нет, а не что они потеряны.
- [x] Формулировки безопасны и не вызывают паники.
- [ ] При отсутствии данных по пациенту указано, кто и как может их добавить.
- [ ] Отдельный empty state для критичных разделов (назначения, результаты, документы).

## 11. Мобильные empty states

- [x] Текст читаем без зума.
- [x] Кнопка действия удобно нажимается пальцем.
- [x] Нет перегруженности мелкими деталями.
- [ ] Sticky CTA для ключевого действия.
- [ ] Соответствие mobile-паттернам приложения.

## 12. Правило применения

- Агент использует все пункты `[x]` как стандарт для пустых состояний.
- Пункты `[ ]` предлагает как дополнительные улучшения.
- Если пользователь просит “минимальный интерфейс”, агент всё равно сохраняет базовый текст и кнопку действия.
- Для медицинских интерфейсов агент уделяет особое внимание разделу 10.

## 13. Фраза для согласования с пользователем

"Могу настроить стандартные empty states: понятные сообщения, почему экран пуст, что делать дальше, кнопки первого действия, отдельные состояния для поиска, первого входа, ошибок и отсутствия прав. Это поможет пользователю не думать, что система сломалась. Применяем?"

# UX-CHECKLIST-ONBOARDING.md

# UX-CHECKLIST-ONBOARDING.md

Чек-лист onboarding-сценариев.
Цель — помочь пользователю начать работу без растерянности.
`[x]` — по умолчанию, `[ ]` — опционально.

## 1. Первый вход в систему

- [x] Понятно, куда пользователь попал и что это за система.
- [x] Есть один очевидный следующий шаг.
- [x] Пустые экраны не выглядят как ошибка.
- [ ] Короткий приветственный экран.
- [ ] Выбор роли или профиля (если нужно).

## 2. Первый сценарий

- [x] Есть понятная “первая задача”, которую можно выполнить.
- [x] Для этой задачи есть понятная кнопка/CTA.
- [ ] Пошаговый мини-мастер (1–3 шага).
- [ ] Подсветка ключевых элементов (guided tour).

## 3. Пояснения и обучение

- [x] Короткие подсказки вместо длинных инструкций.
- [x] Можно пропустить обучение и вернуться к нему позже.
- [ ] Отдельная страница/раздел “как начать”.
- [ ] Видео/анимации только по запросу, а не принудительно.

## 4. Пустые состояния как часть onboarding

- [x] Empty states объясняют, что делать.
- [x] Есть кнопка “создать первый элемент” или аналог.
- [ ] Отдельные тексты для первого входа.
- [ ] Примеры данных или “демо-просмотр”.

## 5. Настройка профиля/предпочтений

- [ ] Минимальный набор обязательных настроек.
- [ ] Остальные настройки можно сделать позже.
- [ ] Нет длинных анкет без реальной необходимости.
- [ ] Для медпроектов — разделение клинических и нефункциональных настроек.

## 6. Возврат к onboarding

- [x] Пользователь может найти onboarding снова.
- [ ] Есть раздел “помощь” / “как пользоваться”.
- [ ] Возможно повторное прохождение guided tour.

## 7. Медицинский контекст

- [x] Не перегружать врача/оператора лишними шагами.
- [x] Onboarding не должен мешать критичным рабочим задачам.
- [ ] Отдельные onboarding-сценарии для роли врача, пациента и администратора.
- [ ] Подсказки, где найти ключевые разделы (назначения, результаты, документы).

## 8. Правило применения

- Агент использует onboarding в новых продуктах и сложных системах.
- Для очень простых сценариев можно ограничиться пустыми состояниями с подсказками.
