# AUDIT-GUIDE

# AUDIT-GUIDE — как проверять проект

> Этот файл нужен для регулярного аудита проекта целиком и по отдельным слоям.

## Когда запускать аудит

- Когда пользователь пишет «проверь проект».
- После крупного этапа работы.
- Перед релизом или передачей проекта.
- Если есть ощущение хаоса или потери контроля.

## Уровень 1 — быстрый аудит

Проверить:

- Понятно ли текущее состояние проекта.
- Обновлены ли `LAYER-3/project-status.md` и `HANDOFF.md`.
- Есть ли следующий шаг.
- Нет ли явных пустых или устаревших документов.

## Уровень 2 — аудит по слоям

### Слой: продукт

Проверить:

- Понятна ли цель проекта.
- Понятно ли для кого он делается.
- Понятен ли текущий этап развития.
- Не расползся ли scope.

### Слой: архитектура

Проверить:

- Описан ли стек.
- Понятны ли основные части системы.
- Нет ли противоречий между архитектурой и реальным состоянием.

### Слой: процесс

Проверить:

- Есть ли понятный workflow.
- Обновляется ли handoff.
- Фиксируются ли уроки проекта.

### Слой: качество

Проверить:

- Есть ли базовый подход к тестированию.
- Понятно ли как проверять новую задачу.
- Нет ли зон, которые никто не проверяет.

### Слой: безопасность

Проверить:

- Понятно ли где лежат чувствительные данные.
- Есть ли правила доступа.
- Есть ли явные риски по ключам, данным, ролям или публикации.

## Уровень 3 — полный аудит

Подготовить отчёт в таком формате:

- Что уже хорошо организовано.
- Что отсутствует или устарело.
- Что противоречит друг другу.
- Какие 3 риска самые важные.
- Какие 3 шага дадут максимальный эффект.

## Формат ответа агента

- Хорошо.
- Не хватает.
- Риски.
- Следующий лучший шаг.

# AUDIT-CHECKLIST — Чек-лист проверки пакета в AI-сервисе

> Используй этот чек-лист, чтобы убедиться что пакет документации работает корректно
> при переносе в новый AI-инструмент (ChatGPT, Gemini, Grok, Copilot и др.).
> Проверяй пункт за пунктом — от структуры до поведения агента.

---

## Блок 1 — Структура пакета (проверяй до запуска агента)

### Обязательные файлы
- [ ] `README.md` — существует и описывает назначение пакета
- [ ] `CLAUDE.md` или аналог для твоей среды — существует и заполнен
- [ ] `llms.txt` — существует (точка входа для агента)
- [ ] `LAYER-1/workflow.md` — существует, описывает уровни разработки
- [ ] `HANDOFF.md` — **если новый проект**: содержит статус `template` или пустые поля в `[]`
- [ ] `HANDOFF.md` — **если активный проект**: заполнен реальными данными (не шаблон)
- [ ] `LAYER-1/scope-guard.md` — существует
- [ ] `LAYER-3/project-status.md` — существует (для уровней 1 и 2)

### Папки
- [ ] `LAYER-2/` — содержит ключевые шаблоны ТЗ (discovery/specs/ux/qa)
- [ ] `LAYER-3/` — существует (для уровней 1 и 2)
- [ ] `tasks/` — существует (хотя бы пустая)

---

## Блок 2 — Запуск новой сессии (тест автозапуска)

Открой любой AI-чат, скопируй содержимое `CLAUDE.md` как системный промпт (или загрузи репо).
Напиши агенту: **«Начнём»** и проверь:

- [ ] Агент прочитал `HANDOFF.md` **автоматически** (не нужно было просить)
- [ ] Агент прочитал `LAYER-3/project-status.md`
- [ ] Агент выдал краткое резюме (3–5 строк): где мы, что делали, что дальше
- [ ] Агент предложил 1–3 варианта следующего шага
- [ ] Агент **не начал сразу писать код** без плана

**Если `HANDOFF.md` — чистый шаблон (новый проект):**
- [ ] Агент запустил конвейер `llms.txt` / `LAYER-1/workflow.md`
- [ ] Агент предложил выбрать уровень (0 / 1 / 2)
- [ ] Агент **не двинулся дальше** без явного ответа о выборе уровня

---

## Блок 3 — Маршрутизация по задачам

Проверяй каждый пункт отдельным сообщением агенту:

| Что написать агенту | Что должен сделать агент | OK? |
|---|---|---|
| «Как работает процесс разработки?» | Прочитать `LAYER-1/workflow.md` | |
| «Что добавить дальше в проект?» | Обратиться к `LAYER-1/feature-radar.md` | |
| «Выбери стек для нового проекта» | Обратиться к `LAYER-1/stack-presets.md` | |
| «Есть ошибка, всё сломалось» | Прочитать `LAYER-1/testing-guide.md` и `LAYER-1/error-handling.md` | |
| «Давай заодно добавим новую функцию» (в середине задачи) | Назвать тип и предложить Backlog | |
| «Откатись к предыдущей версии» | Использовать `LAYER-1/error-handling.md` | |
| «Проверь здоровье проекта» | Запустить `LAYER-1/audit.md` или `audit-agent` | |

---

## Блок 4 — Защита от scope creep

Напиши агенту в середине любой задачи: **«А давай ещё добавим авторизацию»**

- [ ] Агент остановился и не начал реализацию сразу
- [ ] Агент назвал тип запроса (🔴 OUT OF SCOPE / 🟡 SCOPE EXPANSION)
- [ ] Агент спросил: добавить в Backlog или это критично сейчас?
- [ ] Агент **не внёс запись в ROADMAP.md** без твоего явного «да»

---

## Блок 5 — Автосохранение контекста

Выполни любую небольшую задачу (например: «Создай задачу TASK-001»). После завершения:

- [ ] Агент обновил `HANDOFF.md` **автоматически**
- [ ] Агент обновил `LAYER-3/project-status.md`
- [ ] Агент сказал «Контекст сохранён ✅»
- [ ] Агент обновил нужный доп. файл (fixes/features/lessons) по таблице из `CLAUDE.md`

**Проверка триггера напоминания:**
- [ ] После ~10 правок файлов без сохранения агент написал предупреждение ⚠️

---

## Блок 6 — Стиль общения

Проверяй на реальном диалоге:

- [ ] Агент задаёт **один вопрос за раз** (не список из 5)
- [ ] Агент объясняет технические термины простым языком
- [ ] Агент не двигается дальше без явного «всё верно» / «да» / «поехали»
- [ ] Агент показывает план (2–5 шагов) перед выполнением
- [ ] Агент делает резюме после каждого блока работы

---

## Блок 7 — Уровни разработки (если новый проект)

**Уровень 0 — Быстрый старт:**
- [ ] Агент задал **один** вопрос об идее
- [ ] Агент предложил минимальный стек (3 строки: интерфейс / данные / хостинг)
- [ ] Создал только: `CLAUDE.md`, `HANDOFF.md`, `LAYER-3/project-status.md`, `tasks/TASK-001.md`
- [ ] Сразу перешёл к написанию кода после «да»

**Уровень 1 — Стандартный MVP:**
- [ ] Агент провёл интервью по блокам из `LAYER-2/discovery/project-interview.md`
- [ ] После каждого блока — мини-резюме и ожидание подтверждения
- [ ] Документы создавались по очереди с подтверждением
- [ ] Код появился только на Этапе 8

**Уровень 2 — Production MVP:**
- [ ] На Этапах 1–2 никаких файлов не создавалось
- [ ] Код не появился до завершения Этапа 8
- [ ] Каждый документ создавался отдельным шагом

---

## Блок 8 — Конфликты и крайние случаи

Это критические сценарии из аудита — проверяй отдельно:

- [ ] **Тест шаблона vs активного проекта:** скопируй шаблонный HANDOFF.md в новый проект → агент должен запустить llms.txt / LAYER-1/workflow.md, а не продолжить «с прошлого места»
- [ ] **Тест потери контекста:** спроси агента «Ты помнишь что мы делали?» после новой сессии без HANDOFF → агент должен предложить `LAYER-1/context-recovery.md`
- [ ] **Тест конфликта инструкций:** если задача в `tasks/` противоречит `LAYER-2/` — агент должен выбрать `tasks/` (высший приоритет)
- [ ] **Тест несозданных файлов (Level 0):** агент не должен падать с ошибкой при попытке обновить `LAYER-3/fixes.md` если файл ещё не создан — должен создать его сам

> Role: project-record
> Status: not-started
> Filled by: agent
> Needs approval: no

> Это простой светофор состояния проекта.
> Агент обновляет этот файл после каждого значимого изменения.
> Не требует подтверждения пользователя, но должен отражать реальность.

## Шкала

- 🟢 Хорошо — слой понятен и поддерживается.
- 🟡 Есть пробелы — слой в целом живой, но требует внимания.
- 🔴 Риск — слой слабый или почти не описан.

## Правило светофора для слоя Discovery

| Условие | Индикатор |
|---|---|
| `discovery/INTERVIEW-SUMMARY.md` = `accepted` | 🟢 Хорошо |
| `discovery/INTERVIEW-SUMMARY.md` = `draft` или `pending-approval` | 🟡 Есть пробелы |
| Файл отсутствует или `not-started` | 🔴 Риск |

> Связка: если `INTERVIEW-SUMMARY.md = accepted` в `PROJECT-STATUS.md`,
> слой **Discovery** (Продукт) получает 🟢 в таблице слоёв ниже.
