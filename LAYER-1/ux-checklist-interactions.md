# UX-CHECKLIST-NOTIFICATIONS.md

# UX-CHECKLIST-NOTIFICATIONS.md

Чек-лист уведомлений для агента.
Цель — не потерять важные события и не утопить пользователя в шуме.
`[x]` — по умолчанию, `[!]` — обязательно, `[ ]` — опционально.

## 1. Типы уведомлений

- [x] Уведомления об успехе действия (success).
- [x] Уведомления об ошибке (error).
- [x] Предупреждения (warning).
- [x] Информационные уведомления (info).
- [ ] Напоминания (reminder).
- [ ] Системные уведомления (обновления, регламент).

## 2. Каналы уведомлений

- [!] Inline-уведомления в контексте экрана.
- [x] Временные всплывающие уведомления (toast).
- [ ] Панель уведомлений/инбокс.
- [ ] E-mail / внешние каналы (фиксируются отдельно).

## 3. Правила приоритета

- [!] Критичные уведомления не должны теряться среди неважных.
- [x] Критичные события показываются ближе к центру внимания (верх экрана, центр).
- [x] Второстепенные события могут быть только в виде toast или в панели уведомлений.
- [ ] Возможность просмотреть историю важных уведомлений.

## 4. Success-уведомления

- [x] Понятно написано, что именно успешно сделали.
- [x] Нет лишнего шума (кратко, по делу).
- [x] Минимальный визуальный акцент (зелёный/успешный статус).
- [ ] Ссылка "посмотреть результат" или "перейти к записи".

## 5. Error-уведомления

- [!] Объясняют, что произошло, простыми словами.
- [!] Подсказывают, что делать дальше.
- [!] Понятно, где именно ошибка (форма, сеть, доступ).
- [x] Для ошибок формы — сообщение возле поля.
- [x] Для системных ошибок — понятное общее сообщение.
- [ ] Код ошибки/детали — отдельно, для поддержки.

## 6. Warning-уведомления

- [x] Обозначают потенциальную проблему, но не блокируют работу.
- [x] Дают возможность пользователю осознанно продолжить или изменить действие.
- [ ] Можно скрыть предупреждение, если оно не критично.
- [ ] Отдельная визуальная маркировка (жёлтый/оранжевый).

## 7. Info-уведомления

- [x] Используются редко и по делу.
- [x] Не маскируют критичные уведомления.
- [ ] Содержат короткий совет или факт.
- [ ] Возможность скрыть и не показывать снова.

## 8. Напоминания

- [ ] Есть явный список событий, по которым допустимы напоминания.
- [ ] Напоминания не мешают критичным действиям.
- [ ] Пользователь может отключить или настраивать часть напоминаний.
- [ ] Для медицинских сценариев напоминания сформулированы максимально нейтрально и точно.

## 9. Визуальная иерархия

- [x] Критичные уведомления визуально заметнее.
- [x] Цвета и иконки согласованы между экранами.
- [x] Текст не мельче основного текста в интерфейсе.
- [ ] Есть единый стиль для всех уведомлений в продукте.

## 10. Медицинский контекст

- [!] Формулировки не вызывают лишней тревоги при некритичных событиях.
- [!] Критичные медицинские события выделены явно.
- [!] Уведомления не создают ощущение “данные потеряны”, если это не так.
- [ ] Для пациента и врача используются разные формулировки, соответствующие роли.
- [ ] Важные уведомления можно найти повторно (история).

## 11. Правило применения

- Агент использует `[!]` как обязательные требования.
- Агент применяет `[x]` как стандарт по умолчанию.
- `[ ]` предлагает как дополнительные опции под сценарий.

# UX-CHECKLIST-TABLES-AND-BULK-ACTIONS.md

# UX-CHECKLIST-TABLES-AND-BULK-ACTIONS.md

Чек-лист таблиц и массовых действий.
`[x]` — по умолчанию, `[ ]` — опционально.

## 1. Таблицы

- [x] Понятные заголовки колонок.
- [x] Важные колонки расположены ближе к началу.
- [x] Основное действие по строке понятно (открыть/посмотреть).
- [ ] Возможность настроить видимые колонки.
- [ ] Закреплённые колонки (например, имя, ID).

## 2. Массовые действия

- [ ] Выбор нескольких строк (чекбокс).
- [ ] Панель действий для выбранных элементов.
- [ ] Подтверждение перед опасными массовыми действиями.
- [ ] Понятно, сколько элементов выбрано.

## 3. Пагинация и навигация

- [x] Пагинация там, где данных много.
- [x] Видно, на какой странице находится пользователь.
- [ ] Выбор размера страницы.
- [ ] Кнопка “перейти к первой/последней странице”.

## 4. Поиск и фильтры в таблицах

- [x] При необходимости — поиск по ключевым полям.
- [x] Фильтры по важным полям.
- [ ] Сохранённые фильтры.
- [ ] Быстрые фильтры (по статусу, дате и т.п.).

## 5. Empty state таблиц

- [x] Пояснение, почему нет строк.
- [x] Подсказка, как добавить данные или изменить фильтры.
- [ ] Отдельное состояние для “ничего не найдено” и “данных ещё нет”.

## 6. Медицинский контекст

- [x] Таблицы с пациентами/случаями не должны давать ощущение, что данные “пропали”.
- [ ] Для критичных таблиц отдельные фильтры по рискам/статусам.
- [ ] Важно, чтобы массовые действия не могли случайно “спрятать” или удалить важные записи без подтверждения.

## 7. Правило применения

- Агент использует таблицы там, где много однотипных данных.
- Массовые действия даёт только там, где это безопасно и реально экономит время.

# UX-CHECKLIST-SEARCH-FILTERS.md

# UX-CHECKLIST-SEARCH-FILTERS.md

Чек-лист поиска и фильтров.
Цель — сделать поиск и фильтрацию данных понятными и предсказуемыми.
`[x]` — по умолчанию, `[ ]` — опционально.

## 1. Поиск

- [x] Поиск там, где пользователь ожидает его увидеть.
- [x] Понятное поле с placeholder (что можно искать).
- [x] Чёткое поведение: поиск по Enter или по вводу.
- [ ] Подсказки по поисковым запросам.
- [ ] Автодополнение.

## 2. Фильтры

- [x] Только реально нужные фильтры, без перегруза.
- [x] Понятные названия фильтров.
- [x] Видно, какие фильтры сейчас активны.
- [ ] Панель фильтров, которую можно свернуть.
- [ ] Сохранённые наборы фильтров.

## 3. Сортировка

- [x] Сортировка по 1–2 главным колонкам.
- [x] Понятно, по какому полю сейчас идёт сортировка.
- [ ] Возможность менять направление сортировки.
- [ ] Сохранение последней сортировки для пользователя.

## 4. Сброс параметров

- [!] Есть простой способ сбросить фильтры и поиск.
- [x] Вариант “сбросить всё” не прячется.
- [ ] Кнопка “вернуться к рекомендациям по умолчанию”.

## 5. Empty state с фильтрами

- [x] Если ничего не найдено — есть текст “ничего не найдено”.
- [x] Есть подсказка “измените фильтры или запрос”.
- [x] Есть кнопка “сбросить фильтры” или аналог.
- [ ] Примеры запросов, которые дадут результаты.

## 6. Мобильный контекст

- [x] Фильтры не занимают весь экран постоянно.
- [x] Есть кнопка открытия/закрытия фильтров.
- [ ] Sticky кнопка “фильтры”/“поиск”.
- [ ] Bottom sheet для фильтров.

## 7. Медицинский контекст

- [x] Фильтры и поиск не должны приводить к “пропаже” пациентов/документов без понятного объяснения.
- [x] При критичных списках (пациенты, случаи, результаты) должно быть понятно, что список ограничен фильтрами.
- [ ] Отдельные фильтры для приоритетных статусов (например, “критичные результаты”).
- [ ] Явная подсветка активных фильтров в таких списках.

## 8. Правило применения

- Агент включает минимальный поиск/фильтры везде, где есть большие списки.
- Расширенные фильтры добавляются только при реальной потребности, а не “на всякий случай”.

# UX-CHECKLIST-HISTORY-AND-AUDIT.md

Чек-лист истории и аудита действий.
Особенно важен для медицинских, юридических и B2B-систем.

- [x] Видно, что было изменено.

- [ ] Возможность отфильтровать по типу изменений.

## 2. Кто что сделал

- [!] Видно, кто внёс изменение (пользователь/роль).
- [x] Для автоматических событий указан источник (система, интеграция).
- [ ] Сводная история по пользователю.

## 3. Интерфейс истории

- [x] История читаема: упорядочена по времени.
- [x] Важные события выделены визуально.
- [ ] Возможность разворачивать подробности.
- [ ] Фильтры и поиск по истории.

## 4. Медицинский контекст

- [!] История видна там, где изменение влияет на пациента/данные.

- [ ] Ограничение доступа к части истории по ролям.

## 5. Правило применения

- Агент добавляет историю изменений к сущностям, где важно “кто и что поменял”.

# UX-CHECKLIST-MICROCOPY.md

# UX-CHECKLIST-MICROCOPY.md

Чек-лист микротекста (microcopy).
`[x]` — по умолчанию, `[ ]` — опционально.

## 1. Общий стиль

- [x] Короткие фразы.
- [x] Понятный язык без лишнего жаргона.
- [x] От первого лица системы или нейтрально (без путаницы).
- [ ] Согласованный tone of voice для всего продукта.

## 2. Кнопки и действия

- [x] Подписи кнопок описывают действие (“Сохранить”, “Отправить”, а не “Ок”).
- [x] Обратные действия понятны (“Отменить”, “Вернуться”).
- [ ] Тексты на кнопках согласованы across продукт.

## 3. Ошибки и предупреждения

- [x] Ошибка объясняет, что не так и что делать.
- [x] Без обвинения пользователя.
- [x] Без технического шума, если он не нужен.
- [ ] Отдельные тексты для разных типов ошибок (форма, сеть, доступ).

## 4. Пустые состояния и onboarding

- [x] Microcopy в empty states помогает начать, а не просто констатирует факт.
- [ ] Тон дружелюбный, но не назойливый.
- [ ] Для медицинских экранов — аккуратный, спокойный язык.

## 5. Медицинский контекст

- [x] Нет чрезмерно тревожных формулировок там, где ситуация не критична.
- [x] Критичные ситуации обозначены ясно и уважительно.
- [ ] Отдельные шаблоны текстов для пациента и для врача.

## 6. Правило применения

- Агент использует понятный, короткий microcopy как стандарт.
- При сомнении между “официальным” и “человеческим” текстом — выбирать более понятный, но корректный.

# UX-CHECKLIST-INDEX.md

# UX-CHECKLIST-INDEX.md

Оглавление UX-чек-листов для проекта.
Помогает агенту и человеку выбрать нужный набор под конкретный проект.

## 0. Точка входа для агента

- `UX-CHECKLIST-STARTER-FLOW.md`  
  Агент запускает этот сценарий в начале каждой новой проектной беседы.
  Определяет тип проекта, платформу и сложность — и предлагает один готовый UX-режим
  из `UX-PRESET-MODES.md`. Пользователь отвечает просто: «да» или «упрости».

## 1. Базовые чек-листы

Эти чек-листы составляют основу большинства проектов.

- `UX-CHECKLIST-DEFAULT.md`  
  Стандартный набор UX-решений для типового web-интерфейса (структура экранов, формы, списки, empty states, ошибки).

- `UX-CHECKLIST-MINIMAL.md`  
  Упрощённый интерфейс без лишних блоков, когда нужен максимально простой UI.

- `UX-CHECKLIST-ACCESSIBILITY.md`  
  Базовая доступность: контраст, клавиатура, screen reader, мобильная доступность.

- `UX-CHECKLIST-MOBILE.md`  
  Mobile-first и мобильные интерфейсы: touch targets, навигация, формы, ошибки.

- `UX-CHECKLIST-EMPTY-STATES.md`  
  Пустые состояния: “нет данных”, “ничего не найдено”, первый вход, ошибка загрузки.

- `UX-CHECKLIST-ONBOARDING.md`  
  Первый вход пользователя: приветствие, первый шаг, пустые состояния как часть онбординга,
  настройка профиля. Медицинский контекст — не мешать клиническим задачам.

## 2. Домены и роли

Эти чек-листы включаются при специфических типах систем.

- `UX-CHECKLIST-MEDICAL.md`  
  Для медицинских и около-медицинских проектов: безопасность, снижение риска ошибок, ясные статусы.

- `UX-CHECKLIST-ROLE-BASED-UX.md`  
  Для систем с разными ролями (врач, пациент, администратор, оператор и т.п.).

- `UX-CHECKLIST-PERMISSIONS-AND-ACCESS.md`  
  Интерфейс прав и доступа: понятные сообщения об ограничениях, видимость ролей,
  ошибки доступа. Особенно важен для медицинских систем — избыточный доступ такой же
  риск, как его отсутствие.

- `UX-CHECKLIST-DASHBOARDS.md`  
  Для дашбордов и обзорных экранов: KPI, статусы, очереди, критичные события.

## 3. Состояния и взаимодействия

- `UX-CHECKLIST-NOTIFICATIONS.md`  
  Уведомления: success, error, warning, info, напоминания, приоритеты.

- `UX-CHECKLIST-TABLES-AND-BULK-ACTIONS.md`  
  Таблицы и массовые действия: списки, пагинация, выбор нескольких элементов.

- `UX-CHECKLIST-SEARCH-FILTERS.md`  
  Поиск, фильтры, сортировка, сброс параметров, поведение “ничего не найдено”.

- `UX-CHECKLIST-HISTORY-AND-AUDIT.md`  
  История изменений и аудит: кто что сделал и когда, особенно важно для медицины и B2B.

## 4. Поддержка и тексты

- `UX-CHECKLIST-HELP-AND-SUPPORT.md`  
  Помощь, центр справки, связь с поддержкой.

- `UX-CHECKLIST-MICROCOPY.md`  
  Микротекст: кнопки, ошибки, подсказки, тон общения.

## 5. Как агент выбирает чек-листы под проект

Агент должен использовать этот файл совместно с `UX-CHECKLIST-LEGEND.md`.

### 5.1. По типу проекта

- Простой кабинет / MVP:
  - Обязательно: `DEFAULT`, `MINIMAL`, `ACCESSIBILITY`, `EMPTY-STATES`.
  - При необходимости: `MOBILE`, `SEARCH-FILTERS`, `TABLES-AND-BULK-ACTIONS`.

- Внутренний сервис / админка / B2B:
  - Обязательно: `DEFAULT`, `ACCESSIBILITY`, `EMPTY-STATES`, `SEARCH-FILTERS`, `TABLES-AND-BULK-ACTIONS`.
  - Часто нужно: `DASHBOARDS`, `HISTORY-AND-AUDIT`, `NOTIFICATIONS`.

- Медицинский/околомедицинский сервис:
  - Обязательно: `MEDICAL`, `ACCESSIBILITY`, `EMPTY-STATES`, `NOTIFICATIONS`, `HISTORY-AND-AUDIT`, `ROLE-BASED-UX`.
  - При наличии дашбордов: `DASHBOARDS`.
  - При мобильном сценарии: `MOBILE`.

- Публичный личный кабинет / пациентский портал:
  - Обязательно: `DEFAULT`, `MOBILE`, `ACCESSIBILITY`, `EMPTY-STATES`.
  - Для медицины: плюс `MEDICAL`, `ROLE-BASED-UX`, `NOTIFICATIONS`.

### 5.2. По уровню сложности

- Минимальный интерфейс:
  - `UX-CHECKLIST-MINIMAL.md`
  - `UX-CHECKLIST-EMPTY-STATES.md`
  - `UX-CHECKLIST-ACCESSIBILITY.md`

- Продвинутый интерфейс:
  - Всё из Default-блока + доменные и специализированные чек-листы по необходимости.

Агент должен явно указывать, какой набор чек-листов он использует для данного проекта.

# UX-CHECKLIST-STARTER-FLOW.md

Правило старта беседы для выбора UX-режима.

## 1. Когда запускать

Агент запускает этот сценарий:
- в начале новой проектной беседы;
- при старте нового модуля;
- если UX-режим ещё не был выбран.

## 2. Цель

Не заставлять пользователя выбирать отдельные UX-чек-листы вручную.
Вместо этого агент сам предлагает один рекомендованный UX-режим.

## 3. Что агент определяет в начале

Агент быстро определяет:

1. Тип проекта:
- обычный кабинет / MVP;
- внутренний сервис / админка;
- медицинский / около-медицинский сервис;
- пациентский кабинет / портал;
- data-heavy система;
- dashboard-ориентированный продукт.

2. Платформу:
- веб;
- mobile-first;
- смешанный сценарий.

3. Уровень сложности:
- минимальный;
- стандартный;
- расширенный.

4. Есть ли:
- чувствительные данные;
- несколько ролей;
- большие списки и фильтры;
- обзорные панели и KPI.

## 4. Что агент делает после этого

Агент выбирает режим из файла `UX-PRESET-MODES.md`.

### 4.1. Базовый режим
Для стандартных проектов.

### 4.2. Минимальный режим
Если нужен простой интерфейс без перегруза.

### 4.3. Медицинский режим
Если проект связан с медициной или чувствительными данными.

### 4.4. Mobile-first режим
Если основной сценарий происходит на мобильном устройстве.

### 4.5. Data-heavy режим
Если в проекте много списков, реестров, таблиц и фильтров.

### 4.6. Dashboard режим
Если важен обзор состояния системы, KPI, приоритетов и очередей.

## 5. Как агент формулирует предложение

Агент должен предлагать не список файлов, а один режим.

Пример 1:
“Для этого проекта рекомендую базовый UX-режим.
Он уже включает нормальную структуру экранов, пустые состояния, ошибки, формы и доступность.
Если хотите, могу упростить его до минимального.”

Пример 2:
“Для этого проекта рекомендую медицинский UX-режим.
Он уже включает безопасные сценарии, понятные статусы, историю изменений, уведомления, role-based UX, доступность и пустые состояния.
При необходимости добавлю mobile-first расширение.”

Пример 3:
“Для этого проекта рекомендую data-heavy режим.
Он уже включает поиск, фильтры, таблицы, пустые состояния и базовую доступность.
Если нужен overview-экран, добавлю dashboard-режим.”

## 6. Что пользователь может ответить

Пользователю не нужно знать имена файлов.
Он может ответить просто:

- “Да, применяем.”
- “Сделай минимальный вариант.”
- “Добавь mobile-first.”
- “Без dashboard.”
- “Оставь только базу.”
- “Добавь медицинские требования.”

## 7. Что делает агент после согласия

После согласия агент:
- фиксирует выбранный режим;
- использует чек-листы, входящие в него;
- строит интерфейс по этим стандартам;
- не заставляет пользователя заново выбирать UX-элементы на каждом шаге.

## 8. Если режим не выбран

Если пользователь не подтвердил режим:
- агент использует `UX-CHECKLIST-FALLBACK-RULES.md`;
- перед созданием нового раздела предлагает краткий список подходящих UX-элементов для конкретного экрана.

## 9. Главный принцип

На старте беседы пользователь должен видеть не сложную систему документов, а понятную рекомендацию:
“Вот подходящий UX-режим для вашего проекта, он уже включает всё основное.”
