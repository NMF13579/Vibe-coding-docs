## Базовый порядок (модульный)

Этот порядок базовый. Если этап не нужен в текущем запросе, его можно пропустить по согласованию с владельцем.

1. Получить задачу
2. Прочитать ROADMAP — соответствует ли текущему этапу?
   *(если ROADMAP ещё не создан — пропустить этот шаг; ROADMAP создаётся в рамках llms.txt / LAYER-1/workflow.md)*
3. Если используется Git и это крупная задача — предложить сделать коммит текущего состояния как точку возврата
4. Составить план
5. Показать план владельцу
6. Написать: «Подтвердите выполнение плана.»
7. Дождаться: «да» / «выполняй» / «go»
8. Писать код

> **Молчание ≠ согласие.** Если владелец не ответил на план — один раз мягко переспросить:
> «Жду подтверждения — продолжать по плану выше?»
> Если ответа снова нет — оставаться на паузе. Начинать работу без явного «да» запрещено.

## Перед реализацией экрана

Если задача затрагивает **новый экран** или **заметно меняет** то, что видит пользователь (не одна правка формулировки при неизменной структуре):

1. В `LAYER-1/workflow.md` есть шаг процесса, которому экран соответствует (или явно записано, что экран вне процесса — и зачем он тогда нужен).
2. В `LAYER-2/ux/SCREEN-MAP.md` есть этот экран (или понятный блок навигации к нему).
3. В `LAYER-2/ux/WIREFRAMES.md` есть схема экрана (блоки сверху вниз или отсылка «как у экрана X»).
4. В `LAYER-2/ux/` согласована цепочка записей: `PAGES.md` и по смыслу `TEMPLATES.md`, `ORGANISMS.md`, `MOLECULES.md`, `ATOMS.md` — настолько, чтобы было ясно, что на экране и из чего он собран (порядок см. `LAYER-2/ux/ATOMIC-DECOMPOSITION.md`).

Если чего-то из перечисленного **нет** — **не** писать код и **не** заводить в `tasks/` задачу на реализацию этого экрана; сначала дописать документацию и согласовать с владельцем **коротким человеческим резюме** (без внутреннего жаргона).

*Исключение:* правка только текста или мелочи без изменения шагов, полей, блоков или сценария — при условии, что след в документах для этого экрана уже был.

## Формат плана

```
## План: [название]

**Цель:** [что должно работать после]
**Этап ROADMAP:** [этап X]
**Затронутые файлы:** [список]

**Шаги:**
1. [действие]
2. [действие]

**Риски:** [что может пойти не так]
**Не трогаю:** [что остаётся без изменений]

Подтвердите выполнение плана.
```

```
- [файл] — [что изменено]

**Как проверить:**
```

## Прерывание плана (откат или экстренная ситуация)

Если в процессе выполнения плана пользователь пишет «откати», «всё сломалось», «стоп»:
1. Немедленно остановить выполнение
2. Зафиксировать в HANDOFF.md: на каком шаге остановились и что было сделано
3. Перейти к `LAYER-1/error-handling.md`
4. После стабилизации — вернуться к плану с того места, где остановились

> Незавершённый план лучше потерянного контекста.

## Конфликт задач

Если владелец даёт новую задачу, пока предыдущая находится в статусе 🟡 В работе:

1. **Не бросать** текущую задачу молча
2. Написать владельцу:
   «⚠️ Сейчас активна TASK-XXX (статус: В работе, шаг [N]).
   Если переключаемся — фиксирую прогресс в HANDOFF.md и перехожу к новой задаче.
   Если продолжаем — новую задачу добавляю в очередь после текущей.
   Что делаем?»
3. Дождаться явного выбора владельца
4. При переключении: записать в `HANDOFF.md` текущий шаг + статус + что осталось
5. Только потом открывать новую задачу

> Параллельное выполнение двух задач одним агентом запрещено.
> Исключение: задачи явно независимы и владелец написал «делай параллельно».

## Когда останавливаться

- Задача затрагивает данные пользователей или auth
- Нужно удалить, переименовать таблицу или поле в БД
- Предлагается новая внешняя зависимость
- Задача выходит за рамки ROADMAP
- Непонятно что хочет владелец

Формула: «Стоп. Прежде чем продолжить — мне нужно уточнить: [вопрос].»

## Этап 0 — Определение ситуации

Первое, что делает агент — задаёт один вопрос:

> «Привет! Прежде чем начать — скажи:
> это **новый проект** (кода ещё нет) или **уже есть написанный код**,
> и ты хочешь добавить документацию?»

- Если **новый проект** → перейти к выбору уровня ниже.
- Если **есть код** → перейти к конвейеру «Подключение к существующему проекту».

> ⚠️ Агент не движется дальше без ответа на этот вопрос.

## Этап 0б — Выбор уровня разработки (только для новых проектов)

Агент предлагает выбрать уровень:

> «Привет! Прежде чем начать — выбери формат работы:
>
> **Уровень 0 — Быстрый старт** ⚡
> Идея → стек → код. Минимум документов. Подходит для прототипов и экспериментов.
> Время до первого кода: ~15 минут.
>
> **Уровень 1 — Стандартный MVP** 🏗️
> Интервью → документы → план → код. Управляемый процесс с фиксацией решений.
> Подходит для первых реальных продуктов.
> Время до первого кода: ~45 минут.
>
> **Уровень 2 — Production MVP** 🚀
> Полное исследование концепции → модели данных → принципы решений → код.
> Подходит для проектов с реальными пользователями и планом роста.
> Время до первого кода: ~1.5–2 часа.
>
> Какой уровень выбираешь? Или хочешь объяснить подробнее?"

После ответа — перейти к соответствующему конвейеру.

> ⚠️ Агент не движется дальше без явного выбора уровня.

## Повышение уровня

В любой момент проекта владелец может написать:
- «Переходим на уровень выше»
- «Хочу добавить документацию»
- «Проект вырос, нужна структура»

Агент:
1. Читает текущие документы проекта
2. Определяет что уже сделано
3. Составляет список недостающих документов для нового уровня
4. Создаёт их по очереди с подтверждением
5. Обновляет `HANDOFF.md` и `LAYER-3/project-status.md`

Таблица повышения:

| Текущий уровень | Переход | Что добавляется |
|---|---|---|
| 0 → 1 | Добавляем документы | `LAYER-2/discovery/vision.md`, `LAYER-2/specs/architecture.md`, `LAYER-2/specs/roadmap.md`, `LAYER-3/` |
| 1 → 2 | Углубляем документы | `LAYER-2/discovery/vision.md`, `LAYER-2/discovery/mvp-scope.md`, `LAYER-2/ux/USER-FLOWS.md` (+ при необходимости: data-models, decision-principles, project-context — не зафиксированы отдельными файлами в шаблоне) |
| 0 → 2 | Полный переход | Все документы уровней 1 и 2 по очереди |

### Завершение Уровня 0

**Шаг 1 — Мониторинг признаков**

При каждой сессии агент проверяет наличие хотя бы одного признака:

| Признак | Как выглядит |
|---------|-------------|
| Появились реальные пользователи | Кто-то кроме владельца использует продукт |
| Данные требуют защиты | Личные данные, медицинская информация, финансы |
| Задачи зависят друг от друга | «Сначала нужно сделать X, потом Y» |

| Проект показывается другим | Демо, презентация, передача в команду |

Если признак обнаружен — агент говорит:

> «Замечаю что проект вырос: [признак].
> Рекомендую перейти на Уровень 1 — это добавит структуру и защитит от потери прогресса.
> Перейти сейчас или продолжим текущую задачу?»

⚠️ Агент не переходит без явного «да» от владельца.

**Шаг 2 — Подтверждение получено**

Агент проверяет перед переходом:
- [ ] `HANDOFF.md` актуален (отражает последнее состояние)
- [ ] Активная задача имеет STATE: `done` или `in_progress` (не брошена на полуслове)

Если что-то не готово — сначала привести в порядок, потом переходить.

**Шаг 3 — Фиксация в HANDOFF.md**

Агент добавляет запись:

```
Причина перехода: [конкретный признак из таблицы]
Последняя активная задача: TASK-XXX — [STATE]

```

**Шаг 4 — Запуск перехода**

Агент запускает конвейер из раздела «Повышение уровня» (0 → 1).
Новые задачи не создаются до завершения перехода.
> ⚠️ AGENT-BOOTSTRAP.md на Level 0 **опционален** — запускай только если проект неожиданно сломался или контекст потерян. Для обычных сессий достаточно HANDOFF.md.

## Конвейер Уровень 2 — Production MVP 🚀

### Роль агента: единая роль (инженер-исполнитель с фокусом на продукт)

> На этом уровне документация создаётся инкрементально.
> Каждый документ — отдельный шаг с подтверждением.
> Код не пишется до завершения Этапа 8.

**Этап 1 — Контекст платформы (только устно)**

Агент уточняет:
- Какие платформы используются (Lovable, Supabase, и т.д.)
- Что предоставляется из коробки
- Что означают «прототип», «MVP», «production» для этого проекта

Документы не создаются.

**Этап 2 — Исследование концепции (без документов)**

Агент ведёт расширенное интервью на основе `LAYER-2/discovery/project-interview.md`:
- Блоки 0–4 из `project-interview.md` как основа
- На Level 2 добавляются вопросы глубже: B2B/B2C, конкурентный контекст, модели монетизации
- Агент фиксирует ответы в журнал `LAYER-3/interview-session.md` (см. `LAYER-2/discovery/project-interview.md`)

Файлы пока не создаются. Переход к Этапу 3 только после явного «всё верно» от владельца.

**Этап 3 — Видение продукта**

Создаёт или дополняет `LAYER-2/discovery/vision.md`:
- Проблема и целевые пользователи
- Ценностное предложение
- Позиционирование
- Философия MVP

> Без архитектуры и деталей реализации.

**Этап 4 — Границы MVP**

Создаёт или уточняет `LAYER-2/discovery/mvp-scope.md`:
- Что необходимо для MVP
- Что явно не входит в scope
- Намеренные упрощения
- Критерии приёмки MVP

**Этап 5 — Пользовательские сценарии**

Создаёт `LAYER-2/ux/user-flows.md`:
- Основные пользовательские пути
- B2B и B2C потоки (если применимо)
- Happy path и критические решения

> Без UI-экранов. Только логика и взаимодействие.

**Этап 6 — Модели данных**

Создаёт `LAYER-2/specs/data-models.md`:
- Основные сущности и связи
- Владение и границы
- Концептуальная полнота

> Только концептуальная модель, без SQL.

**Этап 7 — Принципы принятия решений**

Создаёт или дополняет `LAYER-2/specs/decisions.md` (отдельный `decision-principles.md` в каркасе не требуется):
- Как оцениваются компромиссы
- Приоритеты (простота, расширяемость, удобство MVP)
- Порядок разрешения конфликтов

**Этап 8 — Контекст проекта**

Создаёт или обновляет `LAYER-3/PROJECT-MEMORY.md` / `LAYER-3/project-context-draft.md` — компактное резюме всего проекта.
Этот документ создаётся последним.

**Этап 9 — Подготовка к реализации**

1. Создать `CLAUDE.md` (заполненный)
2. Создать системный промпт рабочего пространства
3. Создать `LAYER-3/` с languages.md, integrations.md, project-status.md
4. Создать `LAYER-2/specs/roadmap.md`

### Завершение Уровня 2 — Production и Maintenance Mode

**Шаг 1 — Мониторинг признаков готовности к production**

При каждой сессии агент проверяет наличие хотя бы одного признака:

| Признак | Как выглядит |
|---------|-------------|
| ROADMAP закрыт | Все задачи Phase 1 и Phase 2 имеют STATE: `done` |
| Критерии MVP выполнены | Все пункты из `LAYER-2/discovery/mvp-scope.md` отмечены |
| Продукт стабилен | Нет открытых багов блокирующих основной сценарий |
| Есть реальные пользователи | Продукт используется вне тестового режима |

Если признаки обнаружены — агент говорит:

> «Замечаю что MVP готов: [признак].
> Рекомендую запустить финальный чеклист перед production.
> Перейти к проверке сейчас или есть незакрытые задачи?»

⚠️ Агент не объявляет уровень завершённым без явного «да» от владельца.

**Шаг 2 — Финальная проверка перед production**

Агент запускает `LAYER-1/task-protocol.md` и показывает результат:

```
🚀 Отчёт готовности к production

✅ Выполнено: [список]
⚠️ Не выполнено: [список]
❌ Блокеры: [список или «нет»]

Готов к запуску? Или сначала закроем блокеры?
```

Агент не двигается дальше пока есть пункты ❌.

**Шаг 3 — Подтверждение получено**

Агент проверяет перед закрытием уровня:
- [ ] Все задачи из ROADMAP имеют STATE: `done` или `cancelled` с причиной
- [ ] `RELEASE-CHECKLIST.md` пройден без блокеров
- [ ] `HANDOFF.md` обновлён в эту сессию
- [ ] Документы отражают реальное состояние продукта (не планы)

**Шаг 4 — Фиксация в HANDOFF.md**

Агент добавляет запись:

```
## Production MVP запущен — [дата]

Что построено: [2-3 пункта ключевого функционала]
Закрытые задачи: TASK-001 ... TASK-XXX
Намеренно не вошло в MVP: [из mvp-scope.md]
Известные ограничения: [если есть]
```

**Шаг 5 — Переход в Maintenance Mode**

После фиксации агент переходит в режим поддержки:

> «MVP запущен. Переходим в режим поддержки.
> Теперь каждый новый запрос проходит через `LAYER-1/scope-guard.md` —
> новые функции только через новую Phase в ROADMAP.
> Что делаем первым после запуска?»

В Maintenance Mode агент:
- При каждой сессии читает `HANDOFF.md` + `LAYER-3/project-status.md`
- Новые функции — только через новую Phase в `LAYER-2/specs/roadmap.md`
- Баги — через `LAYER-1/testing-guide.md` → отдельная задача
- Архитектурные изменения — через `LAYER-1/error-handling.md` перед началом
## Конвейер — Подключение к существующему проекту 📋

### Роль агента: единая роль (инженер-исполнитель с режимом диагностики)

> Агент не придумывает — он фиксирует реальность.
> Не предлагает менять стек или архитектуру без запроса.
> Документы описывают то, что есть, а не то, что должно быть.

**Этап 1 — Сканирование**

Агент изучает репозиторий: структуру папок, `package.json` / `requirements.txt` / аналоги, основные файлы.

Показывает краткое резюме:
```
Вижу:
- Стек: [что нашёл]
- Тип продукта: [web-приложение / бот / API / другое]
- Структура: [src/, pages/, и т.д.]

Это верно? Что исправить?
```

Ждёт подтверждения перед движением дальше.

**Этап 2 — Три вопроса о состоянии** (по одному)

1. «Что уже работает — что можно показать или потрогать прямо сейчас?»
2. «Что сейчас сломано или в процессе?»
3. «Что планируется следующим шагом?»

После каждого ответа — короткое резюме, уточнение если нужно.

**Этап 3 — Итоговая картина**

Агент показывает сводку:
```
## Понял так:

**Продукт:** [название / рабочее название]
**Стек:** [список]
**Что работает:** [список]
**В процессе / не работает:** [список]
**Следующий шаг:** [формулировка]

Всё верно?
```

Ждёт «всё верно» перед созданием документов.

**Этап 4 — Создание документов по факту** (каждый → подтверждение)

> Документы создаются на основе реального состояния, не с нуля.

1. `LAYER-2/discovery/vision.md` — на основе того, что уже сделано и для кого
2. `LAYER-2/specs/architecture.md` — на основе найденного стека и структуры
3. `LAYER-3/project-status.md` — текущее реальное состояние
4. `LAYER-3/languages.md` + `integrations.md` — по факту кода
5. `HANDOFF.md` — «где мы сейчас» = точка входа после подключения
6. `LAYER-2/specs/roadmap.md` — на основе «следующего шага» из Этапа 2

`LAYER-1/security.md` — создать если в коде есть auth, данные пользователей или API.

**Этап 5 — Диагностика**
