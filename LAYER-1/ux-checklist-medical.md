# UX-CHECKLIST-MEDICAL.md

# UX-CHECKLIST-MEDICAL.md

Медицинский UX-чек-лист для агента.
Используется для интерфейсов врача, пациента, администратора клиники и других медицинских сценариев.
Пункты `[x]` включаются по умолчанию.
Пункты `[!]` обязательны.
Пункты `[ ]` можно добавить по запросу.

## 1. Общий принцип

- [!] Интерфейс должен снижать риск ошибки пользователя.
- [!] Интерфейс должен быть понятен без технического жаргона.
- [!] Главные данные и действия должны быть видны сразу.
- [!] Количество лишних шагов должно быть минимальным.
- [!] Важные действия должны иметь ясную обратную связь.

## 2. Структура экрана

- [x] Заголовок страницы.
- [x] Короткое описание, что делает экран.
- [!] Один главный сценарий на экран.
- [!] Один главный CTA.
- [!] Вторичные действия не должны отвлекать от основного.
- [ ] Блок клинических подсказок.
- [ ] Блок предупреждений по контексту.

## 3. Навигация

- [x] Простая навигация с понятными названиями.
- [!] Критичные разделы доступны за минимальное число шагов.
- [!] Текущий раздел и контекст пациента/документа видны явно.
- [x] Возврат назад без путаницы.
- [ ] Хлебные крошки.
- [ ] Быстрый переход к частым действиям.

## 4. Ключевые данные

- [!] Главные данные выше второстепенных.
- [!] Важные статусы выделены ясно.
- [!] Статусы не различаются только цветом.
- [!] Единицы измерения и подписи показаны явно.
- [ ] Закреплённый summary пациента/случая.
- [ ] Блок последних критичных событий.

## 5. Формы

- [!] Явные названия полей.
- [!] Обязательные поля отмечены.
- [!] Ошибки показываются рядом с полем.
- [!] Ошибки формулируются понятно и безопасно.
- [!] Логичный порядок полей.
- [!] Связанные поля сгруппированы.
- [!] Опасные действия требуют подтверждения.
- [ ] Подсказки по заполнению.
- [ ] Пошаговые формы для сложных сценариев.
- [ ] Автосохранение черновика.
- [ ] Проверка конфликтов или пропусков данных.

## 6. Ошибки и защита от риска

- [!] Ошибка объясняет, что делать дальше.
- [!] Критичные действия требуют явного подтверждения.
- [!] Нельзя полагаться только на всплывающие уведомления.
- [!] Есть понятное сообщение об успешном завершении.
- [!] Есть безопасный сценарий отмены там, где это возможно.
- [ ] Retry button после ошибки загрузки.
- [ ] Дополнительное предупреждение перед особенно опасными действиями.

## 7. Empty states и отсутствие данных

- [x] Пояснение, почему данных нет.
- [x] Подсказка, что делать дальше.
- [!] Пустой экран не должен выглядеть как поломка системы.
- [ ] Отдельный empty state для нового пациента/нового случая.
- [ ] Пример допустимого формата данных.

## 8. Списки, таблицы, записи

- [x] Понятные подписи колонок.
- [!] Главные клинические поля видны без лишних кликов.
- [!] Основное действие по записи понятно сразу.
- [ ] Поиск.
- [ ] Фильтры.
- [ ] Сортировка.
- [ ] Закрепление важных колонок.
- [ ] Переключение таблица / карточки для мобильной версии.

## 9. Подтверждения и модальные окна

- [!] Подтверждение удаления.
- [!] Подтверждение выхода без сохранения.
- [!] Подтверждение чувствительных действий.
- [!] Понятный заголовок модального окна.
- [!] Должно быть ясно, что произойдёт после подтверждения.
- [ ] Двойное подтверждение для особенно опасных действий.

## 10. Мобильная версия

- [x] Крупные зоны нажатия.
- [!] Главные действия доступны без сложной навигации.
- [!] Экран не перегружен второстепенной информацией.
- [ ] Sticky CTA.
- [ ] Упрощённый мобильный режим для врача на ходу.

## 11. Доступность и читаемость

- [!] Контрастный текст.
- [!] Понятные подписи.
- [!] Логичная структура заголовков и блоков.
- [!] Смысл не передаётся только цветом.
- [!] Критичные статусы различимы визуально и текстом.
- [ ] Режим повышенной читаемости.
- [ ] Увеличенные размеры шрифта для сложных экранов.

## 12. История и прозрачность действий

- [!] Пользователь должен понимать, что он сделал и что произошло.
- [!] После важного действия система должна давать явное подтверждение.
- [ ] История действий пользователя.
- [ ] Таймлайн изменений.
- [ ] Отображение автора и времени изменения.

## 13. Роли и контекст

- [!] Интерфейс должен учитывать роль пользователя.
- [!] Врач, пациент и администратор не должны видеть одинаково перегруженный экран.
- [ ] Role-based dashboard.
- [ ] Role-based shortcuts.
- [ ] Отдельные сценарии для врача и пациента.

## 14. Правило применения

- Агент использует все пункты `[!]` как обязательные.
- Агент использует пункты `[x]` как стандартный медицинский UX-набор.
- Пункты `[ ]` предлагает как улучшения под конкретный сценарий.
- Если есть конфликт между красотой и безопасностью, агент выбирает безопасность и ясность.

# UX-CHECKLIST-ROLE-BASED-UX.md

# UX-CHECKLIST-ROLE-BASED-UX.md

Чек-лист role-based UX.
Нужен, когда у системы несколько типов пользователей.
`[x]` — по умолчанию, `[!]` — обязательно, `[ ]` — опционально.

## 1. Определение ролей

- [!] Роли и основные типы пользователей определены.
- [x] Для каждой роли описаны основные цели и задачи.
- [ ] Зафиксированы ограничения по доступу и функциям.

## 2. Интерфейсы по ролям

- [x] Начальные экраны различаются для разных ролей (если нужно).
- [x] Меню и дашборды адаптированы под роль.
- [!] Пользователь не видит лишнего.
- [ ] Отдельные onboarding-сценарии по ролям.

## 3. Навигация по ролям

- [x] Врач/оператор/администратор видит релевантные разделы.
- [x] Не показываются пункты, к которым нет доступа (или ясно помечены).
- [ ] Быстрые ссылки на ключевые задачи для каждой роли.

## 4. Медицинский контекст

- [!] Врач видит клинический контекст и нужные инструменты.
- [!] Пациент видит упрощённый и безопасный интерфейс.
- [!] Администратор видит настройки и отчёты, но не лишние клинические детали.
- [ ] Для каждого типа пользователя есть пример основного сценария.

## 5. Правило применения

- Агент проектирует интерфейс, исходя из роли, а не “универсальной” страницы для всех.

# UX-CHECKLIST-PERMISSIONS-AND-ACCESS.md

# UX-CHECKLIST-PERMISSIONS-AND-ACCESS.md

Чек-лист интерфейсов прав и доступа.
`[x]` — по умолчанию, `[!]` — обязательно, `[ ]` — опционально.

## 1. Видимость и понятность

- [!] Пользователь понимает, к чему у него есть доступ.
- [!] Нет “тихих отказов” (кнопка есть, но ничего не происходит).
- [x] При отсутствии доступа отображается понятное сообщение.
- [ ] Видно, кто может выдать доступ.

## 2. Ошибки доступа

- [!] При попытке сделать запрещённое действие есть явное сообщение.
- [!] Сообщение объясняет, что делать дальше (к кому обратиться или какую роль нужно иметь).
- [x] Ошибки доступа не выглядят как техническая поломка.
- [ ] Отдельные тексты для разных ролей.

## 3. Интерфейс ролей

- [x] Роли описаны понятными словами.
- [x] Для каждой роли обозначены основные возможности.
- [ ] Есть страница/раздел “ролями и доступами управляет …”.
- [ ] Отдельный интерфейс для администратора прав.

## 4. Медицинский контекст

- [!] Доступ к чувствительным данным разделён по ролям.
- [!] Врач, пациент, администратор не видят лишнего.
- [!] Из интерфейса понятно, почему доступ ограничен.
- [ ] Дополнительные предупреждения при выдаче расширенных прав.
- [ ] Логирование изменений прав доступа.

## 5. Правило применения

- Агент всегда добавляет понятные сообщения об отсутствии доступа.
- Для медсистем агент учитывает, что избыточный доступ — такой же риск, как и отсутствие доступа.

# UX-CHECKLIST-DASHBOARDS.md

# UX-CHECKLIST-DASHBOARDS.md

Чек-лист дашбордов и обзорных экранов.
Цель — дать быстрый обзор состояния системы/пациентов/процессов.
`[x]` — по умолчанию, `[ ]` — опционально.

## 1. Общие принципы

- [x] Дашборд отвечает на вопрос: “что сейчас важно?”.
- [x] Вверху — ключевые показатели/статусы.
- [x] Ниже — детали и списки.
- [ ] Нет сложных графиков без необходимости.

## 2. Структура дашборда

- [x] Верхняя полоса KPI/статусов (3–6 блоков).
- [x] Один главный блок со списком/очередью/задачами.
- [x] Дополнительные блоки по приоритету.
- [ ] Отдельный блок рисков/alert’ов.
- [ ] Быстрые действия рядом с ключевыми блоками.

## 3. Блоки KPI

- [x] Каждый блок имеет понятный заголовок.
- [x] Показатель ясен без расшифровки.
- [x] Цветовая кодировка используется аккуратно.
- [ ] Тренд/динамика (стрелка вверх/вниз, мини-график).

## 4. Списки и очереди

- [x] Списки сортируются по приоритету.
- [x] Видно, какие элементы требуют внимания.
- [ ] Фильтры для списка (например, по статусу).
- [ ] Быстрые действия по элементу.

## 5. Визуализация

- [ ] Простые графики (линейный, столбчатый, круговой).
- [ ] Мини-графики (sparklines) внутри KPI-блоков.
- [ ] Подписи и легенды понятны без объяснений.

## 6. Медицинский контекст

- [x] Вверху дашборда — критичные статусы и пациенты.
- [x] Никакой “красоты ради красоты” в ущерб ясности.
- [x] Понятно, какие пациенты/случаи требуют немедленного внимания.
- [ ] Разграничение дашбордов по ролям (врач, руководитель, администратор).
- [ ] Отдельный блок “критические результаты/алерты”.

## 7. Мобильный дашборд

- [x] KPI-блоки читаемы и легко тапаются.
- [x] Очереди/списки адаптированы под прокрутку.
- [ ] Приоритет — сверху, остальное ниже.
- [ ] Возможно, отдельный компактный mobile-dashboard.

## 8. Правило применения

- Агент делает дашборд только там, где нужен обзор, а не просто список.
- В медицине дашборд должен помогать быстро видеть, кому/чему уделить внимание в первую очередь.
