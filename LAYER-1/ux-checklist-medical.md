# UX-CHECKLIST-MEDICAL.md

# UX-CHECKLIST-MEDICAL.md

Медицинский UX-чек-лист для агента.  
Базовые UX-пункты (форма, навигация, empty states, общие ошибки) берутся из `ux-checklist-core.md`; здесь остаётся только клиническая специфика.

## ⚖️ Комплаенс 152-ФЗ

- [!] До сбора медицинских ПДн показывается согласие и ссылка на политику.
- [!] Пользователь видит цель обработки и объём собираемых данных в понятной формулировке.
- [!] В интерфейсе есть явный сценарий отзыва согласия и удаления/блокировки данных.
- [!] Критичные действия с ПДн подтверждаются и попадают в журнал.
- [!] Для реализации юридических текстов и форм используется [`LEGAL-152FZ.md`](./LEGAL-152FZ.md).

## 1. Клиническая безопасность интерфейса

- [!] Критичные клинические показатели всегда выше второстепенных данных.
- [!] Статусы риска и срочности передаются не только цветом, но и текстом.
- [!] Любое рискованное действие врача требует явного подтверждения.
- [!] Ошибки формулируются безопасно: что случилось и что делать дальше без двусмысленности.

## 2. Медицинский контекст пациента/случая

- [!] На экране всегда видны идентификатор случая, текущий статус и ответственный.
- [!] Единицы измерения и референсные интервалы отображаются рядом с показателями.
- [ ] Есть закреплённый клинический summary пациента/случая.
- [ ] Есть блок последних критичных событий по пациенту.

## 3. Трассируемость врачебных действий

- [!] Видно, кто и когда внёс клиническое решение или изменение назначения.
- [!] После критичного действия пользователь получает явное подтверждение результата.
- [ ] Доступен таймлайн изменений по пациенту/случаю.
- [ ] Доступен аудит действий по ролям.

## 4. ⚕️ Клинические сценарии

- [!] Валидация дозировок: значение, единицы и допустимый диапазон проверяются до сохранения.
- [!] Журналирование врачебных решений: фиксируется основание решения и ответственный.
- [!] Телемедицинское информированное согласие: согласие подтверждается до консультации и хранится с отметкой времени.
- [!] Приоритизация симптомов: красные флаги отображаются первыми и требуют отдельного подтверждения действия.
- [!] Доступность для пациентов с ОВЗ: крупный шрифт, контраст, понятные формулировки, поддержка экранных ассистентов.

## 5. Ролевой медицинский UX

- [!] Интерфейсы врача, пациента и администратора разделены по целям и доступам.
- [!] Пользователь не видит клинические данные, которые не нужны его роли.
- [ ] Для каждой роли есть отдельный сценарий первого входа.

## 6. Правило применения

- Сначала применять `ux-checklist-core.md`, затем этот документ как медицинское усиление.
- При конфликте «скорость vs клиническая безопасность» всегда приоритет у безопасности.
- Для данных пациента и телемедицины обязательно сверять решения с [`LEGAL-152FZ.md`](./LEGAL-152FZ.md).

# UX-CHECKLIST-ROLE-BASED-UX.md

# UX-CHECKLIST-ROLE-BASED-UX.md

Чек-лист role-based UX.
Нужен, когда у системы несколько типов пользователей.
`[x]` — по умолчанию, `[!]` — обязательно, `[ ]` — опционально.

## 1. Определение ролей

- [!] Роли и основные типы пользователей определены.
- [x] Для каждой роли описаны основные цели и задачи.
- [ ] Зафиксированы ограничения по доступу и функциям.

## 2. Интерфейсы по ролям

- [x] Начальные экраны различаются для разных ролей (если нужно).
- [x] Меню и дашборды адаптированы под роль.
- [!] Пользователь не видит лишнего.
- [ ] Отдельные onboarding-сценарии по ролям.

## 3. Навигация по ролям

- [x] Врач/оператор/администратор видит релевантные разделы.
- [x] Не показываются пункты, к которым нет доступа (или ясно помечены).
- [ ] Быстрые ссылки на ключевые задачи для каждой роли.

## 4. Медицинский контекст

- [!] Врач видит клинический контекст и нужные инструменты.
- [!] Пациент видит упрощённый и безопасный интерфейс.
- [!] Администратор видит настройки и отчёты, но не лишние клинические детали.
- [ ] Для каждого типа пользователя есть пример основного сценария.

## 5. Правило применения

- Агент проектирует интерфейс, исходя из роли, а не "универсальной" страницы для всех.

# UX-CHECKLIST-PERMISSIONS-AND-ACCESS.md

# UX-CHECKLIST-PERMISSIONS-AND-ACCESS.md

Чек-лист интерфейсов прав и доступа.
`[x]` — по умолчанию, `[!]` — обязательно, `[ ]` — опционально.

## 1. Видимость и понятность

- [!] Пользователь понимает, к чему у него есть доступ.
- [!] Нет "тихих отказов" (кнопка есть, но ничего не происходит).
- [x] При отсутствии доступа отображается понятное сообщение.
- [ ] Видно, кто может выдать доступ.

## 2. Ошибки доступа

- [!] При попытке сделать запрещённое действие есть явное сообщение.
- [!] Сообщение объясняет, что делать дальше (к кому обратиться или какую роль нужно иметь).
- [x] Ошибки доступа не выглядят как техническая поломка.
- [ ] Отдельные тексты для разных ролей.

## 3. Интерфейс ролей

- [x] Роли описаны понятными словами.
- [x] Для каждой роли обозначены основные возможности.
- [ ] Есть страница/раздел "ролями и доступами управляет …".
- [ ] Отдельный интерфейс для администратора прав.

## 4. Медицинский контекст

- [!] Доступ к чувствительным данным разделён по ролям.
- [!] Врач, пациент, администратор не видят лишнего.
- [!] Из интерфейса понятно, почему доступ ограничен.
- [ ] Дополнительные предупреждения при выдаче расширенных прав.
- [ ] Логирование изменений прав доступа.

## 5. Правило применения

- Агент всегда добавляет понятные сообщения об отсутствии доступа.
- Для медсистем агент учитывает, что избыточный доступ — такой же риск, как и отсутствие доступа.

# UX-CHECKLIST-DASHBOARDS.md

# UX-CHECKLIST-DASHBOARDS.md

Чек-лист дашбордов и обзорных экранов.
Цель — дать быстрый обзор состояния системы/пациентов/процессов.
`[x]` — по умолчанию, `[ ]` — опционально.

## 1. Общие принципы

- [x] Дашборд отвечает на вопрос: "что сейчас важно?".
- [x] Вверху — ключевые показатели/статусы.
- [x] Ниже — детали и списки.
- [ ] Нет сложных графиков без необходимости.

## 2. Структура дашборда

- [x] Верхняя полоса KPI/статусов (3–6 блоков).
- [x] Один главный блок со списком/очередью/задачами.
- [x] Дополнительные блоки по приоритету.
- [ ] Отдельный блок рисков/alert'ов.
- [ ] Быстрые действия рядом с ключевыми блоками.

## 3. Блоки KPI

- [x] Каждый блок имеет понятный заголовок.
- [x] Показатель ясен без расшифровки.
- [x] Цветовая кодировка используется аккуратно.
- [ ] Тренд/динамика (стрелка вверх/вниз, мини-график).

## 4. Списки и очереди

- [x] Списки сортируются по приоритету.
- [x] Видно, какие элементы требуют внимания.
- [ ] Фильтры для списка (например, по статусу).
- [ ] Быстрые действия по элементу.

## 5. Визуализация

- [ ] Простые графики (линейный, столбчатый, круговой).
- [ ] Мини-графики (sparklines) внутри KPI-блоков.
- [ ] Подписи и легенды понятны без объяснений.

## 6. Медицинский контекст

- [x] Вверху дашборда — критичные статусы и пациенты.
- [x] Никакой "красоты ради красоты" в ущерб ясности.
- [x] Понятно, какие пациенты/случаи требуют немедленного внимания.
- [ ] Разграничение дашбордов по ролям (врач, руководитель, администратор).
- [ ] Отдельный блок "критические результаты/алерты".

## 7. Мобильный дашборд

- [x] KPI-блоки читаемы и легко тапаются.
- [x] Очереди/списки адаптированы под прокрутку.
- [ ] Приоритет — сверху, остальное ниже.
- [ ] Возможно, отдельный компактный mobile-dashboard.

## 8. Правило применения

- Агент делает дашборд только там, где нужен обзор, а не просто список.
- В медицине дашборд должен помогать быстро видеть, кому/чему уделить внимание в первую очередь.
