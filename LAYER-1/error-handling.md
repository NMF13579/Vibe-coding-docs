# ERROR-TYPES

# ERROR-TYPES — Классификация ошибок

> Когда что-то идёт не так — агент называет тип ошибки вслух.
> Это помогает быстро понять причину и выбрать правильное действие.
> Не читать подряд — использовать как справочник при возникновении проблемы.

---

## Типы ошибок

### 🔵 PROCESS ERROR — пропущен этап

**Что это:** агент или владелец пропустили обязательный шаг протокола.

**Примеры:**
- Код написан без подтверждённого плана
- Задача закрыта без прохождения TASK-REVIEW-PROTOCOL
- Новая зависимость добавлена без согласования

**Действие:**
> «Ошибка: PROCESS ERROR. Пропущен шаг: [что именно].
> Возвращаемся и выполняем его сейчас.»

---

### 🟡 LOGIC ERROR — неправильное решение

**Что это:** техническое решение не работает или решает не ту проблему.

**Примеры:**
- Код написан правильно, но делает не то что нужно
- Выбран неподходящий подход к архитектуре
- Запрос к базе возвращает неверные данные

**Действие:**
> «Ошибка: LOGIC ERROR. Решение не соответствует цели задачи.
> Запускаем RETRY RULE. Попытка [1/2]: [новый подход].»

---

### 🟠 CONTEXT ERROR — потеря контекста

**Что это:** агент не знает где мы находимся в проекте или противоречит прошлым решениям.

**Примеры:**
- Агент предлагает то что уже было отклонено
- Агент не знает какая задача активна
- Агент повторяет вопросы на которые уже отвечали

**Действие:**
> «Ошибка: CONTEXT ERROR. Восстанавливаю контекст.»
> → Запустить CONTEXT-LOSS-RECOVERY.md

---

### 🔴 SCOPE ERROR — разрастание задачи

**Что это:** работа выходит за рамки текущей задачи.

**Примеры:**
- В процессе выполнения TASK-003 начали менять код из TASK-001
- Владелец добавляет новые требования в середине задачи
- Агент «улучшает» то что не просили

**Действие:**
> «Ошибка: SCOPE ERROR. Это выходит за рамки TASK-XXX.»
> → Запустить LAYER-1/scope-guard.md

---

### ⚫ BLOCK ERROR — внешний блокер

**Что это:** задача не может быть выполнена из-за внешней зависимости.

**Примеры:**
- Нужен API-ключ которого нет
- Зависит от задачи которая ещё не закрыта
- Нужно решение владельца прежде чем двигаться дальше

**Действие:**
> «Ошибка: BLOCK ERROR. Задача заблокирована: [причина].
> Статус задачи: blocked. Что нужно от тебя: [конкретный вопрос].»
> *(Статус задачи ≠ MODE агента. MODE остаётся текущим.)*

---

## Как агент сообщает об ошибке

Единый формат:

```
[ТИП ОШИБКИ] — [одна строка что произошло]
Причина: [почему это произошло]
Действие: [что делаем дальше]
```

Пример:
```
SCOPE ERROR — запрос на добавление фильтрации в ходе TASK-005
Причина: фильтрация не входит в критерии готовности задачи
Действие: добавляю в Backlog ROADMAP.md, продолжаем TASK-005
```

## Правило для владельца

Если агент называет тип ошибки — это не упрёк, это диагноз.
Каждый тип имеет чёткое действие. Спорить с типом не нужно — нужно выполнить действие.

## Шаг 1 — Определить масштаб поломки

Агент задаёт ОДИН вопрос:

> «Что именно сломалось?
> А) Приложение не открывается совсем
> Б) Отдельная функция не работает
> В) Данные выглядят неправильно
> Г) Что-то другое — опиши»

---

## Шаг 2 — Сценарии отката

### Сценарий А: сломан код (приложение не работает)

**Если используется Git:**
```bash
# Посмотреть последние коммиты
git log --oneline -10

# Откатиться к последнему рабочему коммиту
git revert HEAD           # отменить последний коммит
# или
git checkout <SHA> -- .   # восстановить конкретную версию
```

**Если используется Lovable / Bolt / похожий инструмент:**
- Открыть историю версий в интерфейсе
- Выбрать последнюю рабочую версию
- Нажать «Восстановить»

**Если используется Cursor / Claude Code:**
- Написать агенту: «Покажи что изменилось в последней сессии»
- Агент отменяет только последние изменения, не трогая остальное

---

### Сценарий Б: сломана отдельная функция

1. Агент смотрит `HANDOFF.md` — что делалось в последней сессии
2. Находит изменённые файлы
3. Откатывает только эти файлы
4. Остальной код не трогает

---

### Сценарий В: проблема с базой данных (Supabase)

**Никогда не удалять данные вручную без команды агента.**

Агент проверяет:
- Таблицы → есть ли данные
- Logs → что происходило последние 30 минут
- RLS политики → не заблокированы ли запросы

Если данные повреждены:
- Supabase → Project Settings → Backups → выбрать последний бэкап

---

### Сценарий Г: сломан деплой (сайт недоступен)

1. Проверить статус хостинга (Vercel / Netlify / другой)
2. Откатить последний деплой одной кнопкой в интерфейсе хостинга
3. Убедиться что предыдущая версия работает
4. Только потом чинить причину

## Правила которые предотвращают поломки

| Правило | Почему |
|---------|--------|
| Коммит перед каждой большой задачей | Есть точка возврата |
| Одна задача за раз | Легко найти что сломало |
| Тест после каждого изменения | Ломка обнаруживается сразу (см. `LAYER-1/testing-guide.md` — что и как тестировать) |
| Не менять базу данных без бэкапа | Данные не восстанавливаются |
