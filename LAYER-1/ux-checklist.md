# UX-CHECKLIST-ACCESSIBILITY.md

# UX-CHECKLIST-ACCESSIBILITY.md

Чек-лист доступности (a11y) для агента.
Применяется ко всем интерфейсам, особенно для медицинских и мобильных.
`[x]` — включено по умолчанию, `[!]` — обязательно, `[ ]` — опционально.

## 1. Общие принципы

- [!] Основные сценарии должны быть доступны без мыши.
- [!] Смысл не должен передаваться только цветом.
- [!] Текст должен быть читаем без зума и танцев с бубном.
- [x] Критичные действия и статусы должны быть понятны любому пользователю.
- [ ] Отдельный обзор доступности перед релизом.

## 2. Текст и контраст

- [!] Контраст текста и фона соответствует WCAG AA.
- [!] Основной текст не мельче базового размера (обычно 16 px).
- [x] Второстепенный текст остаётся читаемым.
- [x] Текст на цветных кнопках/бейджах читабелен.
- [x] Нет серого текста на слегка более сером фоне.
- [ ] Проверка контраста для всех ключевых элементов отдельно.

## 3. Цвет и смысл

- [!] Цвет не единственный способ обозначить статус.
- [!] Ошибки и предупреждения имеют и текст, и цвет.
- [x] Важные статусы дополнительно маркируются иконкой или текстом.
- [x] Легенды к цветам в графиках и таблицах понятны.
- [ ] Добавлены текстовые метки к цветовым индикаторам.

## 4. Клавиатурная навигация

- [!] Все основные действия доступны с клавиатуры.
- [!] Фокус по Tab двигается логично и предсказуемо.
- [!] Фокус видно (подсветка, рамка, смена состояния).
- [x] Кнопки и ссылки можно активировать Enter/Space.
- [ ] Есть "перейти к основному контенту" (skip link).
- [ ] Комбинации клавиш для частых действий (где уместно).

## 5. Screen reader и семантика

- [!] Заголовки выстроены по уровню (h1, h2, h3 без хаоса).
- [!] Основные блоки размечены семантически (header, main, nav, footer).
- [!] Интерактивные элементы имеют корректные роли (button, link и т.п.).
- [x] У иконок‑кнопок есть понятный текст (aria-label).
- [x] У картинок есть alt или явное пустое alt, если картинка декоративная.
- [ ] ARIA-атрибуты используются только там, где действительно нужны.

## 6. Формы и ошибки

- [!] Каждое поле имеет подпись (label).
- [!] Ошибка привязана к конкретному полю.
- [!] Ошибки объясняют, что сделать, а не только “неверное значение”.
- [x] Обязательные поля явно отмечены.
- [x] Сообщения об успехе и ошибке понятны на слух.
- [ ] Есть общее summary ошибок для длинных форм.

## 7. Фокус, модальные окна, диалоги

- [!] При открытии модального окна фокус переходит внутрь него.
- [!] При закрытии модального окна фокус возвращается туда, откуда пришёл.
- [!] В модальных окнах есть явная кнопка закрытия.
- [x] Escape закрывает диалог, если это безопасно.
- [x] Пользователь не может “потерять” фокус в невидимой области.
- [ ] Тёмная подложка под модальным окном не мешает работе screen reader.

## 8. Мобильная доступность

- [!] Интерактивные элементы не менее 44x44 px.
- [!] Расстояние между touch‑целями достаточное.
- [!] Основные действия доступны без точного прицеливания.
- [x] Интерфейс корректно работает в портретной ориентации.
- [ ] Поддерживается landscape, если сценарий этого требует.
- [ ] Учитываются системные жесты (без конфликтов).

## 9. Состояния элементов

- [!] Есть состояния: обычное, наведение (если есть), фокус, нажатие, отключено.
- [!] Состояния понятны не только по цвету (форма, иконка, текст).
- [x] Disabled выглядит реально недоступным, но читабельным.
- [x] Загрузка отображается явно (спиннер, skeleton, текст).
- [ ] Важные долгие операции имеют индикатор прогресса.

## 10. Контент и язык

- [!] Формулировки максимально ясные, без лишнего жаргона.
- [!] Длинные фразы разбиты на удобочитаемые блоки.
- [x] Сокращения и аббревиатуры объясняются там, где это критично.
- [x] Важное не прячется в “мелкий серый текст”.
- [ ] Есть глоссарий, если много спецтерминов.

## 11. Медицинский контекст

- [!] Сообщения не вызывают лишней тревоги, если ситуация не критична.
- [!] Важные статусы (например, критические результаты) обозначены ясно.
- [!] Ошибки не порождают ощущение “данные пациента потеряны”.
- [x] Подробные технические детали скрыты за дополнительным действием.
- [ ] Разные роли (врач, пациент, администратор) получают свой уровень детализации.

## 12. Проверка и поддержка

- [x] Есть базовый чек‑лист, по которому агент проходит перед релизом.
- [x] Основные сценарии проверены с клавиатурой.
- [x] Основные сценарии проверены на мобильном устройстве.
- [ ] Проведено отдельное тестирование доступности для медицинских интерфейсов.
- [ ] Запланирован регулярный пересмотр доступности по мере развития продукта.

## 13. Правило применения

- Агент использует все пункты `[!]` как обязательные для любого интерфейса.
- Пункты `[x]` считаются стандартом доступности по умолчанию.
- Пункты `[ ]` агент предлагает как дополнительные меры для повышенной доступности.
- В случае конфликта между визуальным “вау” и доступностью агент выбирает доступность и понятность.

## 14. Фраза для согласования с пользователем

"Могу включить стандартный чек‑лист доступности: контрастный текст, понятные заголовки, работа с клавиатуры, корректные сообщения об ошибках, доступные формы и мобильные touch‑элементы. Обязательные пункты будут учтены во всех экранах. Применяем?"

# UX-CHECKLIST-DASHBOARDS.md

# UX-CHECKLIST-DASHBOARDS.md

Чек-лист дашбордов и обзорных экранов.
Цель — дать быстрый обзор состояния системы/пациентов/процессов.
`[x]` — по умолчанию, `[ ]` — опционально.

## 1. Общие принципы

- [x] Дашборд отвечает на вопрос: “что сейчас важно?”.
- [x] Вверху — ключевые показатели/статусы.
- [x] Ниже — детали и списки.
- [ ] Нет сложных графиков без необходимости.

## 2. Структура дашборда

- [x] Верхняя полоса KPI/статусов (3–6 блоков).
- [x] Один главный блок со списком/очередью/задачами.
- [x] Дополнительные блоки по приоритету.
- [ ] Отдельный блок рисков/alert’ов.
- [ ] Быстрые действия рядом с ключевыми блоками.

## 3. Блоки KPI

- [x] Каждый блок имеет понятный заголовок.
- [x] Показатель ясен без расшифровки.
- [x] Цветовая кодировка используется аккуратно.
- [ ] Тренд/динамика (стрелка вверх/вниз, мини-график).

## 4. Списки и очереди

- [x] Списки сортируются по приоритету.
- [x] Видно, какие элементы требуют внимания.
- [ ] Фильтры для списка (например, по статусу).
- [ ] Быстрые действия по элементу.

## 5. Визуализация

- [ ] Простые графики (линейный, столбчатый, круговой).
- [ ] Мини-графики (sparklines) внутри KPI-блоков.
- [ ] Подписи и легенды понятны без объяснений.

## 6. Медицинский контекст

- [x] Вверху дашборда — критичные статусы и пациенты.
- [x] Никакой “красоты ради красоты” в ущерб ясности.
- [x] Понятно, какие пациенты/случаи требуют немедленного внимания.
- [ ] Разграничение дашбордов по ролям (врач, руководитель, администратор).
- [ ] Отдельный блок “критические результаты/алерты”.

## 7. Мобильный дашборд

- [x] KPI-блоки читаемы и легко тапаются.
- [x] Очереди/списки адаптированы под прокрутку.
- [ ] Приоритет — сверху, остальное ниже.
- [ ] Возможно, отдельный компактный mobile-dashboard.

## 8. Правило применения

- Агент делает дашборд только там, где нужен обзор, а не просто список.
- В медицине дашборд должен помогать быстро видеть, кому/чему уделить внимание в первую очередь.

- [ ] Режим повышенной читаемости.
- [ ] Пошаговые сценарии для критичных процессов.

# UX-CHECKLIST-EMPTY-STATES.md

# UX-CHECKLIST-EMPTY-STATES.md

UX-чек-лист пустых состояний для агента.
Используется для экранов без данных, после фильтра/поиска, при первом входе и при ошибках загрузки.
Пункты `[x]` включаются по умолчанию.
Пункты `[ ]` можно добавить по запросу.

## 1. Общие принципы

- [x] Пустой экран не должен выглядеть как ошибка системы.
- [x] Пользователь должен понимать, что происходит и что делать дальше.
- [x] Empty state — это возможность объяснить, как пользоваться интерфейсом.
- [x] Текст должен быть коротким и понятным.
- [ ] Отдельные empty states для разных контекстов (первый вход, фильтр, ошибка).

## 2. Базовый empty state (нет данных вообще)

- [x] Короткое пояснение, почему здесь пусто.
- [x] Подсказка, что нужно сделать, чтобы появились данные.
- [x] Одна основная кнопка первого действия.
- [ ] Дополнительная ссылка на документацию или помощь.
- [ ] Краткий пример, как выглядит заполненный экран.
- [ ] Ненавязчивая иллюстрация или иконка.

## 3. Empty state после фильтра/поиска

- [x] Сообщение "ничего не найдено".
- [x] Подсказка, что можно изменить фильтр или поиск.
- [x] Кнопка "сбросить фильтры" или аналогичное действие.
- [ ] Предложение часто используемых фильтров или запросов.
- [ ] Примеры запросов, которые дадут результат.
- [ ] Индикатор активных фильтров, который легко сбросить.

## 4. Empty state после удаления данных

- [x] Сообщение, что элементов больше нет.
- [x] Понятный контекст: что именно было удалено.
- [x] Подсказка, можно ли и как создать новый элемент.
- [ ] Временное уведомление о том, что данные удалены.
- [ ] Ссылка на историю изменений или лог, если он есть.

## 5. Empty state при первом использовании (onboarding)

- [x] Сообщение, что это первый вход или пустой старт.
- [x] Подсказка, как начать работать.
- [x] Одна главная кнопка “начать” / “создать первый элемент”.
- [ ] Короткая инструкция шаг за шагом (1–2 шага).
- [ ] Ссылка на подробную инструкцию или обучение.
- [ ] Пример заполненного состояния для ориентира.

## 6. Empty state из-за отсутствия прав доступа

- [x] Сообщение, что у пользователя нет доступа к этой информации.
- [x] Пояснение без технического жаргона.
- [x] Подсказка, к кому обратиться для получения доступа.
- [ ] Кнопка "запросить доступ" или аналогичное действие.
- [ ] Ссылка на политику доступа или роли.

## 7. Empty state при ошибке загрузки (состояние после ошибки, а не только тост)

- [x] Сообщение, что произошла ошибка при загрузке данных.
- [x] Пояснение, что это может быть временная проблема.
- [x] Кнопка “повторить попытку”.
- [ ] Короткий технический текст для поддержки (код ошибки, если уместно).
- [ ] Ссылка на статус сервиса или контакт поддержки.

## 8. Тон и текст

- [x] Текст короткий и конкретный.
- [x] Без обвинения пользователя.
- [x] Без агрессивного юмора или сарказма.
- [x] Формулировки вежливые и спокойные.
- [ ] Адаптация тона под аудиторию (пациент, врач, администратор, внутренний пользователь).

## 9. Визуальное оформление

- [x] Пустое состояние визуально отличается от “сломавшегося” экрана.
- [x] Основной фокус на тексте и кнопке действия.
- [x] Нет лишнего визуального шума.
- [ ] Ненавязчивая иллюстрация/иконка, поддерживающая смысл.
- [ ] Одинаковый визуальный стиль empty states по всему продукту.

## 10. Медицинский контекст (если проект медицинский)

- [x] Empty state не должен создавать ощущение, что “данные пациента исчезли”.
- [x] Явно указано, что данных ещё нет, а не что они потеряны.
- [x] Формулировки безопасны и не вызывают паники.
- [ ] При отсутствии данных по пациенту указано, кто и как может их добавить.
- [ ] Отдельный empty state для критичных разделов (назначения, результаты, документы).

## 11. Мобильные empty states

- [x] Текст читаем без зума.
- [x] Кнопка действия удобно нажимается пальцем.
- [x] Нет перегруженности мелкими деталями.
- [ ] Sticky CTA для ключевого действия.
- [ ] Соответствие mobile-паттернам приложения.

## 12. Правило применения

- Агент использует все пункты `[x]` как стандарт для пустых состояний.
- Пункты `[ ]` предлагает как дополнительные улучшения.
- Если пользователь просит “минимальный интерфейс”, агент всё равно сохраняет базовый текст и кнопку действия.
- Для медицинских интерфейсов агент уделяет особое внимание разделу 10.

## 13. Фраза для согласования с пользователем

"Могу настроить стандартные empty states: понятные сообщения, почему экран пуст, что делать дальше, кнопки первого действия, отдельные состояния для поиска, первого входа, ошибок и отсутствия прав. Это поможет пользователю не думать, что система сломалась. Применяем?"

# UX-CHECKLIST-FALLBACK-RULES.md

# UX-CHECKLIST-FALLBACK-RULES.md

Правило fallback для агента, если UX-чек-листы не были выбраны в начале проекта.

## 1. Когда применять это правило

Агент применяет это правило, если:
- пользователь не выбрал UX-чек-листы на старте проекта;
- пользователь просит создать новый раздел, страницу, экран, блок или сценарий;
- у агента нет явного указания, какой UX-стандарт использовать.

В этом случае агент не должен сразу собирать интерфейс без согласования UX-основы.

## 2. Что агент делает сначала

Перед созданием нового раздела агент должен предложить пользователю короткий список подходящих UX-элементов для этого экрана.

Для каждого элемента агент должен указать:
- что это такое;
- для какой цели применяется;
- является ли это стандартом индустрии;
- включить ли его в текущий экран по умолчанию или только по запросу.

## 3. Формат предложения

Агент показывает список в понятном виде:

- Название элемента.
- Для чего нужен.
- Когда обычно применяется.
- Это индустриальный стандарт или опция.
- Предлагается ли включить по умолчанию.

Пример формата:

- Заголовок и краткое описание экрана — помогают пользователю сразу понять, где он находится и что делать; это стандарт индустрии; рекомендуется включать по умолчанию.
- Empty state — нужен, чтобы пустой экран не выглядел как поломка; это стандарт индустрии для интерфейсов с данными; рекомендуется включать по умолчанию.
- Поиск — нужен для больших списков и каталогов; это стандарт индустрии только для data-heavy разделов; включать, если элементов много.
- Sticky CTA — нужен, если главное действие должно быть всегда под рукой, особенно на мобильном; это полезная практика, но не обязательный стандарт; включать по ситуации.

## 4. Базовый список элементов, которые агент может предлагать

### 4.1. Почти всегда предлагать

- Заголовок экрана.
- Краткое описание или контекст.
- Один главный CTA.
- Empty state.
- Состояние загрузки.
- Ошибка загрузки / ошибка действия.
- Success state после действия.
- Базовая доступность (контраст, понятные подписи, логичная структура).
- Мобильная адаптация, если экран будет использоваться на мобильных устройствах.

### 4.2. Предлагать по ситуации

- Формы.
- Поиск.
- Фильтры.
- Сортировка.
- Таблица или карточки.
- Подтверждение удаления.
- Уведомления.
- История изменений.
- Дашборд/обзорный блок.
- Помощь/подсказки.
- Role-based различия интерфейса.
- Sticky CTA.
- Bottom sheet / модальное окно.
- Bulk actions.
- Audit trail.

## 5. Как агент определяет, что предлагать

Агент должен смотреть на тип раздела:

### 5.1. Если это форма
Предлагать:
- подписи полей;
- обязательные поля;
- ошибки рядом с полем;
- success state;
- подтверждение опасных действий;
- автосохранение или пошаговую форму, если форма длинная.

### 5.2. Если это список или реестр
Предлагать:
- поиск;
- фильтры;
- сортировку;
- empty state;
- состояние “ничего не найдено”;
- пагинацию или lazy load;
- таблицу или карточки;
- bulk actions, если это реально нужно.

### 5.3. Если это detail page / карточка сущности
Предлагать:
- summary-блок сверху;
- смысловые секции;
- главное действие;
- историю изменений;
- блок связанных данных;
- предупреждения, если действия рискованные.

### 5.4. Если это dashboard
Предлагать:
- KPI/статусы;
- приоритетные списки;
- критичные алерты;
- быстрые действия;
- фильтры периода/статуса;
- мобильную компактную версию, если нужно.

### 5.5. Если это медицинский экран
Предлагать дополнительно:
- подтверждения чувствительных действий;
- безопасные формулировки ошибок;
- role-based интерфейс;
- историю изменений;
- понятные статусы;
- приоритетное отображение критичных данных.

## 6. Как агент помечает элементы

Агент должен использовать такие категории:

- **Стандарт индустрии**  
  Элемент считается нормой для таких экранов и обычно включается по умолчанию.

- **Рекомендуемая практика**  
  Элемент не обязателен всегда, но часто полезен и улучшает UX.

- **Опционально**  
  Элемент нужен только в отдельных сценариях.

## 7. Какой ответ агент должен давать пользователю

Перед созданием экрана агент должен кратко писать, например:

“Перед тем как собирать этот раздел, предлагаю выбрать UX-элементы для него:

- Заголовок + краткое описание — стандарт индустрии, помогает понять экран, предлагаю включить.
- Empty state — стандарт индустрии для экранов с данными, предлагаю включить.
- Поиск — стандарт для больших списков, включить если записей будет много.
- Фильтры — рекомендуемая практика для реестров и журналов, включить если нужно быстро сужать выборку.
- История изменений — рекомендуемая практика, особенно полезна в медицинских и административных сценариях.
- Sticky CTA — опционально, полезно на мобильных экранах.

Подтвердите, какие элементы включаем в этот раздел.”

## 8. Приоритет выбора

Если пользователь не хочет выбирать подробно:
- агент автоматически включает все элементы категории “Стандарт индустрии”;
- предлагает 2–4 элемента категории “Рекомендуемая практика”;
- не навязывает “Опционально”.

## 9. Медицинский приоритет

Если проект медицинский или около-медицинский:
- агент должен особенно предлагать безопасность, понятные статусы, empty states, уведомления, историю изменений, role-based UX и подтверждения чувствительных действий;
- если есть сомнение, агент выбирает ясность и безопасность, а не визуальную сложность.

## 10. Главный принцип

Если UX-чек-лист не выбран заранее, агент сначала предлагает набор подходящих UX-элементов для конкретного раздела, а уже потом проектирует интерфейс.
Это защищает проект от случайного дизайна, уменьшает хаос и делает интерфейс более предсказуемым.

  - Обязательно: `DEFAULT`, `MINIMAL`, `ACCESSIBILITY`, `EMPTY-STATES`.

  - Обязательно: `DEFAULT`, `ACCESSIBILITY`, `EMPTY-STATES`, `SEARCH-FILTERS`, `TABLES-AND-BULK-ACTIONS`.
  - Часто нужно: `DASHBOARDS`, `HISTORY-AND-AUDIT`, `NOTIFICATIONS`.

- Медицинский/околомедицинский сервис:

### 5.2. По уровню сложности

- Минимальный интерфейс:
  - `UX-CHECKLIST-MINIMAL.md`

# UX-CHECKLIST-LEGEND.md

# UX-CHECKLIST-LEGEND.md

Легенда и правила использования UX-чек-листов и режимов.

## 1. Обозначения в чек-листах

Во всех UX-чек-листах используются одинаковые отметки:

- `[x]` — включено по умолчанию  
  Элемент считается стандартом для данного чек-листа.  
  Агент может использовать его без дополнительного одобрения, если соответствующий чек-лист активен.

- `[!]` — обязательно  
  Критичный элемент, без которого сценарий считается небезопасным или некорректным.  
  Агент обязан его учесть, если чек-лист активен (особенно в медицинских и высокорискованных сценариях).

- `[ ]` — опционально  
  Дополнительный элемент. Агент может предложить его, но не обязан включать по умолчанию.  
  Включение зависит от контекста проекта и согласования с пользователем.

## 2. UX-режимы вместо ручного выбора чек-листов

Пользователь **не обязан** выбирать отдельные чек-листы.  
Вместо этого используются режимы из `UX-PRESET-MODES.md`:

- Базовый режим — стандартный web-интерфейс.
- Минимальный режим — упрощённый интерфейс/MVP.
- Медицинский режим — медицинские и чувствительные сценарии.
- Mobile-first режим — мобильный приоритет.
- Data-heavy режим — списки, реестры, таблицы, фильтры.
- Dashboard режим — обзорные панели и KPI.

Каждый режим “под капотом” включает набор чек-листов.  
Агент сам подключает нужные чек-листы в зависимости от выбранного режима.

## 3. Как агент действует в начале проекта

В начале новой проектной беседы (или модуля):

1. Агент по нескольким простым вопросам определяет:
   - тип проекта;
   - наличие медицинских/чувствительных данных;
   - mobile-first или нет;
   - есть ли большие списки, таблицы, дашборды;
   - есть ли разные роли пользователей.

2. Агент выбирает один или несколько режимов из `UX-PRESET-MODES.md`.

3. Агент формулирует предложение пользователю **в терминах режима**, а не чек-листов, например:
   - “Рекомендую базовый UX-режим: он уже включает структуру экранов, формы, ошибки, empty states и доступность.”
   - “Так как проект медицинский, предлагаю медицинский UX-режим с историей изменений и безопасными сценариями.”

4. Пользователь может:
   - подтвердить режим как есть;
   - попросить минимальный режим;
   - попросить добавить/убрать mobile-first, dashboard, data-heavy и т.п.

После подтверждения режима агент считает соответствующие чек-листы активными.

## 4. Как агент использует чек-листы после выбора режима

Когда режим выбран:

- Все элементы с `[!]` из активных чек-листов  
  — обязательны к учёту в проекте.

- Все элементы с `[x]`  
  — используются как стандартный набор без дополнительного согласования.

- Элементы с `[ ]`  
  — агент предлагает как улучшения, когда это уместно, особенно в сложных сценариях.

Агент **не обязан** каждый раз перечислять чек-листы пользователю — ему достаточно следовать им при проектировании интерфейса.

## 5. Если режим не был подтверждён

Если в начале работы UX-режим не выбран или пользователь не ответил:

- Агент использует правила из `UX-CHECKLIST-FALLBACK-RULES.md`.
- Перед созданием **каждого нового раздела** агент:
  - предлагает краткий список подходящих UX-элементов (заголовок, empty states, ошибки, таблица, поиск, история и т.п.);
  - для каждого элемента поясняет:
    - для чего он;
    - является ли он стандартом индустрии или опцией;
  - просит подтвердить, что включать.

Это fallback-режим “по заботе” — когда глобальный UX-набор не зафиксирован на старте, но локально про UX всё равно не забывают.

## 6. Приоритеты выбора

Если возникает конфликт:

- Между “красотой” и доступностью  
  → приоритет у доступности.

- Между “минимализмом” и безопасностью (особенно в медицине)  
  → приоритет у безопасности и предсказуемости.

- Между скоростью внедрения и ясностью интерфейса  
  → приоритет у ясности для пользователя.

## 7. Что важно для медицинских проектов

Для медицинских и около-медицинских проектов агент должен:

- Использовать **медицинский режим** как основной.
- Обязательно активировать чек-листы:
  - `UX-CHECKLIST-MEDICAL.md`
  - `UX-CHECKLIST-ACCESSIBILITY.md`
  - `UX-CHECKLIST-EMPTY-STATES.md`
  - `UX-CHECKLIST-NOTIFICATIONS.md`
  - `UX-CHECKLIST-HISTORY-AND-AUDIT.md`
  - `UX-CHECKLIST-ROLE-BASED-UX.md`

- При необходимости добавить:
  - `UX-CHECKLIST-MOBILE.md` — если есть мобильные сценарии;
  - `UX-CHECKLIST-DASHBOARDS.md` — если есть обзорные панели.

Если пользователь явно не отказывается от медицинского режима, агент обязан ориентироваться на повышенную безопасность и ясность интерфейса.

## 8. Краткое правило для агента

- Сначала выбери **режим**, а не отдельные чек-листы.
- Объясни режим пользователю человеческим языком.
- После согласия следуй чек-листам, входящим в этот режим.
- Если режим не выбран — используй fallback и явно предлагаy UX-элементы при создании разделов.

# UX-CHECKLIST-STARTER-FLOW.md

# UX-CHECKLIST-STARTER-FLOW.md

Правило старта беседы для выбора UX-режима.

## 1. Когда запускать

Агент запускает этот сценарий:
- в начале новой проектной беседы;
- при старте нового модуля;
- если UX-режим ещё не был выбран.

## 2. Цель

Не заставлять пользователя выбирать отдельные UX-чек-листы вручную.
Вместо этого агент сам предлагает один рекомендованный UX-режим.

## 3. Что агент определяет в начале

Агент быстро определяет:

1. Тип проекта:
- обычный кабинет / MVP;
- внутренний сервис / админка;
- медицинский / около-медицинский сервис;
- пациентский кабинет / портал;
- data-heavy система;
- dashboard-ориентированный продукт.

2. Платформу:
- веб;
- mobile-first;
- смешанный сценарий.

3. Уровень сложности:
- минимальный;
- стандартный;
- расширенный.

4. Есть ли:
- чувствительные данные;
- несколько ролей;
- большие списки и фильтры;
- обзорные панели и KPI.

## 4. Что агент делает после этого

Агент выбирает режим из файла `UX-PRESET-MODES.md`.

### 4.1. Базовый режим
Для стандартных проектов.

### 4.2. Минимальный режим
Если нужен простой интерфейс без перегруза.

### 4.3. Медицинский режим
Если проект связан с медициной или чувствительными данными.

### 4.4. Mobile-first режим
Если основной сценарий происходит на мобильном устройстве.

### 4.5. Data-heavy режим
Если в проекте много списков, реестров, таблиц и фильтров.

### 4.6. Dashboard режим
Если важен обзор состояния системы, KPI, приоритетов и очередей.

## 5. Как агент формулирует предложение

Агент должен предлагать не список файлов, а один режим.

Пример 1:
“Для этого проекта рекомендую базовый UX-режим.
Он уже включает нормальную структуру экранов, пустые состояния, ошибки, формы и доступность.
Если хотите, могу упростить его до минимального.”

Пример 2:
“Для этого проекта рекомендую медицинский UX-режим.
Он уже включает безопасные сценарии, понятные статусы, историю изменений, уведомления, role-based UX, доступность и пустые состояния.
При необходимости добавлю mobile-first расширение.”

Пример 3:
“Для этого проекта рекомендую data-heavy режим.
Он уже включает поиск, фильтры, таблицы, пустые состояния и базовую доступность.
Если нужен overview-экран, добавлю dashboard-режим.”

## 6. Что пользователь может ответить

Пользователю не нужно знать имена файлов.
Он может ответить просто:

- “Да, применяем.”
- “Сделай минимальный вариант.”
- “Добавь mobile-first.”
- “Без dashboard.”
- “Оставь только базу.”
- “Добавь медицинские требования.”

## 7. Что делает агент после согласия

После согласия агент:
- фиксирует выбранный режим;
- использует чек-листы, входящие в него;
- строит интерфейс по этим стандартам;
- не заставляет пользователя заново выбирать UX-элементы на каждом шаге.

## 8. Если режим не выбран

Если пользователь не подтвердил режим:
- агент использует `UX-CHECKLIST-FALLBACK-RULES.md`;
- перед созданием нового раздела предлагает краткий список подходящих UX-элементов для конкретного экрана.

## 9. Главный принцип

На старте беседы пользователь должен видеть не сложную систему документов, а понятную рекомендацию:
“Вот подходящий UX-режим для вашего проекта, он уже включает всё основное.”

# UX-CHECKLIST-TABLES-AND-BULK-ACTIONS.md

Чек-лист таблиц и массовых действий.
`[x]` — по умолчанию, `[ ]` — опционально.

## 1. Таблицы

- [x] Понятные заголовки колонок.
- [x] Важные колонки расположены ближе к началу.
- [x] Основное действие по строке понятно (открыть/посмотреть).
- [ ] Возможность настроить видимые колонки.
- [ ] Закреплённые колонки (например, имя, ID).

## 2. Массовые действия

- [ ] Выбор нескольких строк (чекбокс).
- [ ] Панель действий для выбранных элементов.
- [ ] Подтверждение перед опасными массовыми действиями.
- [ ] Понятно, сколько элементов выбрано.

## 3. Пагинация и навигация

- [x] Пагинация там, где данных много.
- [x] Видно, на какой странице находится пользователь.
- [ ] Выбор размера страницы.
- [ ] Кнопка “перейти к первой/последней странице”.

## 4. Поиск и фильтры в таблицах

- [x] При необходимости — поиск по ключевым полям.
- [x] Фильтры по важным полям.
- [ ] Сохранённые фильтры.
- [ ] Быстрые фильтры (по статусу, дате и т.п.).

## 5. Empty state таблиц

- [x] Пояснение, почему нет строк.
- [x] Подсказка, как добавить данные или изменить фильтры.
- [ ] Отдельное состояние для “ничего не найдено” и “данных ещё нет”.

## 6. Медицинский контекст

- [x] Таблицы с пациентами/случаями не должны давать ощущение, что данные “пропали”.
- [ ] Для критичных таблиц отдельные фильтры по рискам/статусам.
- [ ] Важно, чтобы массовые действия не могли случайно “спрятать” или удалить важные записи без подтверждения.

## 7. Правило применения

- Агент использует таблицы там, где много однотипных данных.
- Массовые действия даёт только там, где это безопасно и реально экономит время.
