# DEPLOY

# DEPLOY.md

Универсальная инструкция для AI-агента и владельца проекта по безопасному деплою в разные среды: Vercel, Timeweb Cloud, Railway, Render, Docker VPS и другие.

## Цель документа

Этот файл нужен, чтобы деплой можно было запускать без отдельного DevOps-специалиста.  
AI-агент должен читать этот документ перед любым релизом и действовать строго по нему.

## Для каких сред подходит

- Vercel.
- Timeweb Cloud.
- Railway.
- Render.
- Docker на VPS.
- Любая платформа, где есть staging и production или их аналоги.

## Общий принцип

Деплой всегда идёт по маршруту:

1. Локальная проверка.
2. Деплой в тестовую среду.
3. Проверка после деплоя.
4. Только потом деплой в production.
5. После релиза — мониторинг и готовность к откату.

## Окружения

### Local

Локальная среда разработчика.

Используется для:
- запуска проекта на компьютере;
- первичной проверки;
- ручного тестирования;
- проверки переменных окружения.

### Staging

Тестовая среда, максимально похожая на production.

Используется для:
- проверки перед релизом;
- приёмки владельцем;
- smoke-теста;
- проверки интеграций.

### Production

Боевая среда для реальных пользователей.

Правила:
- деплой только после staging;
- деплой только при понятном плане отката;
- никаких экспериментальных изменений напрямую.

## Кто что может делать

### Разрешено без подтверждения

- Проверять статус сборки.
- Читать логи.
- Деплоить в staging.
- Выполнять smoke-тест после staging.

### Только после подтверждения владельца

- Деплой в production.
- Изменение переменных окружения.
- Применение миграций базы данных.
- Переключение домена.
- Откат production.

### Запрещено

- Удалять production-базу данных.
- Менять секреты без фиксации в документации.
- Выполнять необратимые действия без явного подтверждения.
- Деплоить в production, если staging не проверен.

## Как использовать чеклисты

В проекте есть два разных чеклиста, и у каждого своя роль:

- `LAYER-1/deploy-guide.md` — использовать ПЕРЕД КАЖДЫМ деплоем
  в staging и особенно перед production. Это технический список:
  код, тесты, env‑переменные, миграции, инфраструктура.

- `LAYER-1/task-protocol.md` — использовать ПЕРЕД ПЕРВЫМ запуском
  продукта для реальных пользователей или перед крупным публичным
  релизом. Это продуктовый список: сценарии пользователя, безопасность,
  UX, документация.

Агент обязан:

1. Перед любым деплоем пройти `DEPLOY-CHECKLIST.md`.
2. Перед «боевым» запуском продукта пройти ещё и `RELEASE-CHECKLIST.md`.
3. Остановить production‑деплой, если критичные пункты чеклистов не выполнены.

## Универсальный сценарий деплоя

### Шаг 1. Проверить готовность

Перед любым деплоем нужно убедиться:

- код закоммичен;
- ветка понятна (`main`, `staging`, `release/*`);
- переменные окружения описаны в `ENV.md`;
- есть актуальный `CHANGELOG.md`;
- есть план отката в `ROLLBACK-PROTOCOL.md`;
- последний `HANDOFF.md` не противоречит релизу.

### Шаг 2. Прогнать проверки

Минимум:

- линтер;
- тесты, если есть;
- сборка проекта;
- ручная проверка критического сценария.

### Шаг 3. Деплой в staging

Варианты в зависимости от платформы:

- **Git push** в staging-ветку.
- **CLI-команда** платформы.
- **Кнопка Deploy** в панели.
- **AI-агент через MCP / SSH / dashboard workflow**.

После деплоя агент обязан зафиксировать:
- куда задеплоено;
- ссылку на окружение;
- статус сборки;
- краткий результат проверки.

### Шаг 4. Smoke-test staging

Проверить:

- открывается главная страница;
- работает вход или основной сценарий;
- нет явных 500/404 ошибок;
- интеграции отвечают;
- критические формы работают;
- нет сломанных env-переменных.

### Шаг 5. Подтверждение владельца

Перед production агент обязан спросить подтверждение в явной форме:

> Staging проверен. Деплоить в production? Да / нет.

### Шаг 6. Деплой в production

Production-деплой выполняется только после явного подтверждения.

После завершения зафиксировать:
- время релиза;
- commit / tag;
- адрес production;
- успешность smoke-теста;
- кто подтвердил релиз.

### Шаг 7. Мониторинг после релиза

Первые 15–30 минут проверить:

- доступность сайта или API;
- логи ошибок;
- скорость ответа;
- ключевой пользовательский сценарий;
- алерты в мониторинге.

## Примеры запуска деплоя по типу платформы

### Вариант A. Git-based deploy

Подходит для Vercel, Netlify, части конфигураций Timeweb Cloud, Render, Railway.

Схема:

1. Изменения попадают в GitHub.
2. Push в нужную ветку запускает автодеплой.
3. Платформа собирает проект сама.

### Вариант B. CLI deploy

Подходит для Vercel CLI, Railway CLI, Render CLI, Docker CLI и аналогов.

Схема:

1. Агент запускает команду платформы.
2. Получает статус сборки.
3. Проверяет URL релиза.

### Вариант C. SSH / server deploy

Подходит для VPS, Docker-серверов, кастомных Timeweb-конфигураций.

Схема:

1. Подключение к серверу.
2. Обновление кода.
3. Сборка или перезапуск контейнера.
4. Проверка логов и health-check.

## Что обязательно фиксировать после каждого деплоя

Записать в `CHANGELOG.md`, `HANDOFF.md` или release log:

- дата и время;
- среда: staging / production;
- версия или commit;
- что изменилось;
- результат smoke-теста;
- были ли ошибки;
- был ли откат.

## Универсальные вопросы, которые агент должен задать перед production

- Нужны ли миграции базы данных?
- Нужен ли maintenance window?
- Есть ли риск для действующих пользователей?
- Есть ли команда отката?

### 1. Код и версия

- [ ] Все нужные изменения закоммичены.
- [ ] Понятно, какая ветка деплоится.
- [ ] В `CHANGELOG.md` есть краткая запись о релизе.

### 2. Конфигурация

- [ ] Все обязательные переменные описаны в `ENV.md`.
- [ ] Секреты не попали в git.
- [ ] Конфигурация staging и production различается только там, где это ожидаемо.
- [ ] Домены, callback URL и API endpoints проверены.

- [ ] Проект собирается без ошибок.
- [ ] Линтер проходит или есть осознанное исключение.

- [ ] Нет опасных ручных SQL-команд без подтверждения.

- [ ] Целевая среда доступна.
- [ ] Есть права на деплой.
- [ ] Логи доступны.
- [ ] Health-check endpoint известен.

### 6. После деплоя

- [ ] Известно, кто проводит smoke-тест.
- [ ] Известно, где смотреть мониторинг.
- [ ] Известно, как запускать откат.
- [ ] Владелец подтвердил production-релиз.

## Правило остановки

Если хотя бы один критичный пункт не выполнен, production-деплой запрещён.

# ENV

# ENV.md

Универсальное описание переменных окружения для проекта, пригодное для Vercel, Timeweb Cloud, Railway, Render, Docker и VPS.

## Цель документа

Этот файл объясняет:
- какие переменные нужны проекту;
- какие из них обязательные;
- где их задавать;
- какие значения нельзя хранить в git.

## Главные правила

- Никогда не хранить реальные секреты в репозитории.
- В репозитории хранить только `.env.example` без настоящих значений.
- Реальные значения задавать в панели платформы, секрет-хранилище или на сервере.
- Любое изменение env-переменных фиксировать в `HANDOFF.md` и при необходимости в `CHANGELOG.md`.

## Минимальный шаблон категорий переменных

### 1. Системные

Примеры:
- `NODE_ENV`
- `PORT`
- `APP_ENV`

Назначение:
- указывают, в какой среде работает приложение.

### 2. База данных

Примеры:
- `DATABASE_URL`
- `POSTGRES_URL`
- `REDIS_URL`

Назначение:
- подключение к хранилищам данных.

### 3. Аутентификация

Примеры:
- `NEXTAUTH_SECRET`
- `JWT_SECRET`
- `SESSION_SECRET`

Назначение:
- подпись токенов, сессий, логина.

### 4. Внешние API

Примеры:
- `OPENAI_API_KEY`
- `ANTHROPIC_API_KEY`
- `SMTP_PASSWORD`
- `S3_ACCESS_KEY`

Назначение:
- интеграции со сторонними сервисами.

### 5. Публичные переменные

Примеры:
- `NEXT_PUBLIC_API_URL`
- `PUBLIC_APP_NAME`

Правило:
- такие переменные могут попасть во frontend;
- нельзя хранить там секреты.

## Таблица переменных

| Имя | Обязательна | Где используется | Для каких сред | Секрет | Комментарий |
|---|---|---|---|---|---|
| `APP_ENV` | Да | App runtime | local, staging, production | Нет | Значение среды |
| `DATABASE_URL` | Обычно да | Backend / ORM | staging, production | Да | Подключение к БД |
| `AUTH_SECRET` | Обычно да | Auth | staging, production | Да | Секрет сессий |
| `PUBLIC_BASE_URL` | Часто да | Frontend / callbacks | staging, production | Нет | Базовый URL |

## Где задавать переменные

### Local

- файл `.env.local` или аналог;
- не коммитить;
- использовать только для разработки.

### Staging / Production

Можно задавать:
- в Vercel Project Settings;
- в Timeweb Cloud App settings;
- в Railway Variables;
- в Render Environment;
- в Docker Compose / secrets / CI variables;
- в секрет-хранилище.

## Обязательные артефакты рядом

- `.env.example`
- `LAYER-1/deploy-guide.md`
- `LAYER-1/deploy-guide.md`
- `LAYER-1/error-handling.md`

## Что должен делать AI-агент

Перед деплоем агент обязан:

1. Проверить, что все обязательные переменные описаны.
2. Не печатать реальные секреты в чат.
3. Не записывать секреты в git.
4. Уточнить у владельца, если обязательной переменной не хватает.
5. Не продолжать production-деплой, если обязательная переменная отсутствует.

## Минимальный `.env.example`

```env
APP_ENV=local
PUBLIC_BASE_URL=http://localhost:3000
DATABASE_URL=
AUTH_SECRET=
```

# MONITORING

# MONITORING.md

Универсальная инструкция по мониторингу после деплоя для проектов без выделенного DevOps.

## Цель документа

Этот файл нужен, чтобы после релиза быстро понять:
- сервис жив или нет;
- есть ли ошибки;
- нужно ли делать откат.

## Минимально достаточный мониторинг

Даже маленький проект должен иметь 3 уровня наблюдения:

1. **Доступность** — открывается ли сайт или API.
2. **Ошибки** — есть ли 500, падения, исключения.
3. **Ключевой сценарий** — может ли пользователь сделать главное действие.

## Что проверять после каждого деплоя

### Первые 5 минут

- открывается ли приложение;
- отвечает ли health-check;
- нет ли массовых 500/502/503;
- успешна ли авторизация, если она есть.

### Первые 15 минут

- есть ли всплеск ошибок в логах;
- выросло ли время ответа;
- работает ли ключевая интеграция;
- работает ли основной пользовательский сценарий.

### Первый час

- нет ли жалоб пользователей;
- нет ли деградации производительности;
- нет ли повторяющихся ошибок в фоне.

## Что считать красным флагом

Нужно сразу останавливать релиз и готовить откат, если:

- главная страница не открывается;
- логин или основной сценарий сломан;
- наблюдаются массовые ошибки 5xx;
- отсутствует подключение к БД или внешнему API;
- приложение стартовало, но работает не в той среде.

## Рекомендуемые инструменты

Подходят любые аналоги. Выбрать можно один в каждой категории.

### 1. Проверка доступности

Примеры:
- UptimeRobot;
- Better Stack;
- встроенные health-check системы платформы.

### 2. Логи и ошибки

Примеры:
- Sentry;
- встроенные логи Vercel;
- встроенные логи Timeweb Cloud;
- Railway / Render logs;
- Docker logs на сервере.

### 3. Продуктовая аналитика

Примеры:
- PostHog;
- Plausible;
- Umami;
- Simple Analytics.

## Универсальный пост-деплой ритуал

1. Открыть production URL.
2. Проверить health endpoint, если он есть.
3. Выполнить один ключевой сценарий как пользователь.
4. Открыть логи.
5. Открыть ошибки.
6. Наблюдать минимум 15 минут.

## Что должен записывать агент

После релиза агент должен зафиксировать:

- ссылку на релиз;
- была ли ошибка в логах;
- был ли пройден smoke-test;
- нужен ли follow-up;
- нужен ли rollback.

## Критерий успеха релиза

Релиз считается успешным, если:

- приложение доступно;
- основной сценарий работает;
- критичных ошибок нет;
- нет признаков поломки интеграций;
- владелец подтверждает, что всё в порядке.
