# VERCEL-DEPLOY — инструкция для Vercel

> Этот файл дополняет `LAYER-1/deploy-guide.md`.
> Если есть конфликт между ними, при деплое на Vercel
> приоритет у ЭТОГО файла.

---

## 1. Общая схема

На Vercel деплой обычно запускается автоматически:

1. Репозиторий подключен к Vercel.
2. Push в определённую ветку запускает сборку.
3. Vercel создаёт превью (preview) или production‑релиз.

Агент всегда использует общий маршрут из `DEPLOY.md`:
Local → Staging (preview) → Production.

---

## 2. Окружения на Vercel

### 2.1. Preview (аналог staging)

- Любой pull request или push в некритичную ветку.
- URL вида: `https://branch-name-project-name.vercel.app`.
- Используется для:
  - проверки изменений;
  - приёмки владельцем;
  - smoke‑теста.

### 2.2. Production

- Привязан к основной ветке (обычно `main`).
- Имеет основной домен.
- Используется реальными пользователями.

---

## 3. Переменные окружения на Vercel

Все переменные описаны в `LAYER-1/deploy-guide.md`.

На Vercel есть три типа env‑переменных:

- Development — для локальной разработки.
- Preview — для всех preview‑деплоев (staging).
- Production — для продакшена.

Агент обязан:

1. Проверить, что все обязательные переменные описаны в `ENV.md`.
2. Не создавать новых env‑переменных без фиксации в `ENV.md`.
3. Не менять значения production‑переменных без подтверждения владельца.

---

## 4. Деплой в staging (preview)

Вариант A — через Git:

1. Убедиться, что все изменения закоммичены.
2. Создать или обновить ветку (например, `feature/...` или `staging`).
3. Сделать `git push` в эту ветку.
4. Vercel автоматически создаст preview‑деплой.
5. Найти URL превью:
   - в интерфейсе Vercel;
   - или в комментариях к pull request (если настроено).

Вариант B — через Vercel Dashboard:

- Нажать «Deploy» для нужной ветки.
- Дождаться завершения сборки.

После деплоя агент фиксирует:

- URL превью.
- Commit / ветку.
- Статус сборки.

---

## 5. Smoke‑test на Vercel

Агент использует чеклист из `LAYER-1/deploy-guide.md` и сценарии из `LAYER-2/ux/user-flows.md`:

Минимум:

- Открыть preview‑URL.
- Проверить:
  - загрузку главной страницы;
  - регистрацию / вход (если есть);
  - основной пользовательский сценарий;
  - отсутствие явных 500 / 404;
  - корректность работы ключевых интеграций.

---

## 6. Деплой на production

### 6.1. Подготовка

Перед продакшен‑деплоем:

- Пройти `LAYER-1/deploy-guide.md`.
- Убедиться, что preview‑версия проверена и одобрена.
- Получить явное «Да, деплоим в прод» от владельца.

### 6.2. Запуск продакшен‑деплоя

Вариант A — через Git:

- Push в основную ветку (обычно `main`).
- Vercel запускает production‑сборку автоматически.

Вариант B — через интерфейс:

- Открыть Vercel Dashboard.
- Выбрать проект.
- Нажать «Promote to Production» для проверенного preview‑деплоя (если доступно).

После деплоя агент фиксирует:

- время релиза;
- commit / tag;
- production‑URL;
- результат smoke‑теста;
- кто подтвердил релиз.

---

## 7. Мониторинг на Vercel

Используется вместе с `LAYER-1/deploy-guide.md`:

- Открыть вкладку Logs в Vercel.
- Проверить ошибки после релиза.
- Убедиться, что нет всплеска 5xx.

Если настроены внешние инструменты (Sentry, UptimeRobot и т.п.) —
они описываются в `MONITORING.md`.

---

## 8. Откат на Vercel

Если релиз неудачный:

1. Открыть проект в Vercel.
2. Выбрать последнюю стабильную версию из списка Deployments.
3. Нажать «Promote to Production» для этой версии.
4. Убедиться, что на production снова работает стабильная версия.
5. Зафиксировать откат в `CHANGELOG.md` и `LAYER-3/fixes.md`.

Также см. общий `LAYER-1/error-handling.md`.
