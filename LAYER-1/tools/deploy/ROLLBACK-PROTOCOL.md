# ROLLBACK-PROTOCOL — что делать когда всё сломалось

> Этот документ — инструкция на случай катастрофы.
> Агент читает его при сигнале «всё сломалось» / «откати» / «верни как было».

---

## Принцип: сначала стабилизировать, потом разбираться

Не пытаемся чинить сломанное — сначала возвращаем рабочее состояние.

---

## Шаг 1 — Определить масштаб поломки

Агент задаёт ОДИН вопрос:

> «Что именно сломалось?
> А) Приложение не открывается совсем
> Б) Отдельная функция не работает
> В) Данные выглядят неправильно
> Г) Что-то другое — опиши»

---

## Шаг 2 — Сценарии отката

### Сценарий А: сломан код (приложение не работает)

**Если используется Git:**
```bash
# Посмотреть последние коммиты
git log --oneline -10

# Откатиться к последнему рабочему коммиту
git revert HEAD           # отменить последний коммит
# или
git checkout <SHA> -- .   # восстановить конкретную версию
```

**Если используется Lovable / Bolt / похожий инструмент:**
- Открыть историю версий в интерфейсе
- Выбрать последнюю рабочую версию
- Нажать «Восстановить»

**Если используется Cursor / Claude Code:**
- Написать агенту: «Покажи что изменилось в последней сессии»
- Агент отменяет только последние изменения, не трогая остальное

---

### Сценарий Б: сломана отдельная функция

1. Агент смотрит `HANDOFF.md` — что делалось в последней сессии
2. Находит изменённые файлы
3. Откатывает только эти файлы
4. Остальной код не трогает

---

### Сценарий В: проблема с базой данных (Supabase)

**Никогда не удалять данные вручную без команды агента.**

Агент проверяет:
- Таблицы → есть ли данные
- Logs → что происходило последние 30 минут
- RLS политики → не заблокированы ли запросы

Если данные повреждены:
- Supabase → Project Settings → Backups → выбрать последний бэкап

---

### Сценарий Г: сломан деплой (сайт недоступен)

1. Проверить статус хостинга (Vercel / Netlify / другой)
2. Откатить последний деплой одной кнопкой в интерфейсе хостинга
3. Убедиться что предыдущая версия работает
4. Только потом чинить причину

---

## Шаг 3 — Зафиксировать причину

После стабилизации агент записывает в `memory-bank/fixes.md`:
[Дата] — Откат
Причина: [что сломалось]
Что откатили: [какие файлы / коммиты]
Рабочее состояние восстановлено: [да/нет]
Причина поломки: [что стало причиной]
Как не повторить: [что изменить в процессе]

text

## Правила которые предотвращают поломки

| Правило | Почему |
|---------|--------|
| Коммит перед каждой большой задачей | Есть точка возврата |
| Одна задача за раз | Легко найти что сломало |
| Тест после каждого изменения | Ломка обнаруживается сразу |
| Не менять базу данных без бэкапа | Данные не восстанавливаются |

---

## Сигналы для запуска протокола

Агент запускает этот протокол при словах:
- «всё сломалось»
- «откати»
- «верни как было»
- «приложение не работает»
- «белый экран»
- «ошибка 500»
